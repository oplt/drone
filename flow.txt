Warehouse Start Flight + Live Mapping Flow
==========================================

This document traces function names and file paths for:
  A) Start Flight button on the Warehouse page
  B) Live mapping, WebSocket data flow, chunking, and map record persistence


================================================================================
A. START FLIGHT (Warehouse page)
================================================================================

FRONTEND — button click → HTTP
------------------------------

  Step  Function                    File
  ----  --------                    ----
  1     handleStartWarehouseScan    frontend/src/modules/warehouse/views/Warehouse.tsx
  2     startWarehouseScan          frontend/src/modules/warehouse/api/warehouseMissionsApi.ts
  3     setPendingFlightId          frontend/src/modules/mission-runtime/hooks/useMissionRuntime.ts
                                    (via useMissionWebsocketRuntime in Warehouse.tsx)

After launch, Warehouse.tsx also wires:
  - useMissionWebsocketRuntime  → mission status / telemetry WebSocket
  - useAutoStartVideo           → useMissionVideo (camera stream)
  - useWarehouseLiveVoxelMap    → live 3D map WebSocket (once activeFlightId is set)

================================================================================
B. LIVE MAPPING + MAP RECORD + WEBSOCKET + CHUNKING
================================================================================

Two parallel paths:
  1. Live visualization — ROS → bridge → chunk files → WS metadata → frontend download → render
  2. Post-flight record  — ROS rosbag/artifacts → capture session → DB job (persist_capture)


BACKEND — ROS → chunk → WebSocket
---------------------------------

  Role                         Function                              File
  ----                         --------                              ----
  Resolve PointCloud2 topics     resolve_colored_bridge_sources        backend/modules/warehouse/service/live_map_readiness.py
                               probe_live_map_topic_types
  Start colored bridge           start_colored_pointcloud_live_map_bridge
                                                                     backend/modules/warehouse/service/colored_pointcloud_live_map_bridge.py
  ROS callback per cloud         _ColoredPointCloudLiveMapNode._on_pointcloud
                                                                     colored_pointcloud_live_map_bridge.py
  Parse PointCloud2              parse_pointcloud2_msg                 backend/modules/warehouse/service/pointcloud2_parser.py
                               encode_xyzrgb32 / encode_xyz32
  Chunk + publish                _store_and_publish_colored_chunk      colored_pointcloud_live_map_bridge.py
  Write chunk binary             warehouse_live_map_chunk_storage.save_upload
                                                                     backend/modules/warehouse/service/live_map_storage.py
  Write sidecar metadata         save_chunk_metadata                   live_map_storage.py
  Build WS payload               normalize_live_map_payload            backend/modules/warehouse/service/live_map_stream.py

Secondary bridge (odom + nvblox health metadata):

LIVE MAP DATA FLOW (ASCII)
--------------------------

  ROS PointCloud2 topics
    → colored_pointcloud_live_map_bridge._on_pointcloud(msg)
      → live_map_storage.save_upload(chunk bytes)
      → live_map_stream.publish(live_map_update + chunk metadata)
        → api.py websocket_live_map_stream → send_json to WS clients
          → useWarehouseLiveVoxelMap (onMessage)
            → useLiveMapChunkCache sees new chunk.url in changed_chunks
              → fetchWarehouseLiveChunk GET /live-map/{flight}/chunks/{id}/download
                → WarehouseLiveVoxelScene renders CachedLiveMapChunk
