## B. Live mapping + map record + WebSocket + chunking

There are **two parallel paths**:
1. **Live visualization** — ROS → bridge → chunk files → WS metadata → frontend download → Three.js render
2. **Post-flight record** — ROS rosbag/artifacts → capture session → DB job (`persist_capture`)

---

### Map record (persisted scan job, not live WS)

Live chunks on disk under the flight ID are **separate** from the final scanned-map job. Recording flow:

| Step | Function | File |
|------|----------|------|
| Start rosbag / perception session | `port.start_mapping` | perception port via `scan.py` |
| Stop + export wait | `_stop_perception_mapping`, `wait_for_mapping_artifacts` | `scan.py`, `capture_finalize.py` |
| Stage files | `capture_session_service.finalize_session` | `capture.py` |
| Save quality manifest | `build_manifest_from_flight_dir`, `save_flight_manifest` | `live_map_manifest.py` |
| Create DB job + assets | `WarehouseScanMappingService.persist_capture` | `mapping.py` |
| Notify frontend scan done | `warehouse_live_map_stream.finalize` | `live_map_stream.py` |
| Replay saved map later | `build_disk_live_map_snapshot`, `get_scanned_map_live_map_snapshot` | `live_map_replay.py`, `api.py` |
