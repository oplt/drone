What you saw is **not generated from a normal planar / 2D LiDAR alone**. A planar LiDAR only scans one horizontal slice, so it can miss table tops, shelves, hanging objects, tall boxes, or obstacles above/below the scan plane. The Isaac ROS/nvBlox demo uses **depth cameras / stereo cameras and/or 3D LiDAR** to build a **3D voxel map**, then visualizes that map live. Isaac ROS nvBlox is explicitly designed to use **depth cameras and/or 3D LiDAR**, GPU-accelerate 3D reconstruction, and output costmaps for navigation. ([nvidia-isaac-ros.github.io][1])

The basic pipeline is this:

```text
Multi-camera / 3D LiDAR data
        ↓
Depth image or PointCloud2
        ↓
Camera/LiDAR calibration + robot pose / TF
        ↓
Back-project sensor data into 3D points
        ↓
Fuse points into nvBlox voxel map
        ↓
Build TSDF / occupancy / ESDF layers
        ↓
Publish mesh + ESDF slice + Nav2/MoveIt costmap
        ↓
RViz visualization + planner obstacle avoidance
```

In the Isaac ROS graph, nvBlox takes **depth image, color image, and pose** as input, computes a **3D scene reconstruction on the GPU**, slices that reconstruction into a costmap for Nav2, and streams mesh updates into RViz in real time. ([nvidia-isaac-ros.github.io][1])

### What happens technically

Each camera gives a depth image. Every depth pixel means: “at this image coordinate, the surface is X meters away.” Using the camera intrinsics from `CameraInfo`, that pixel is **back-projected into a 3D point** in the camera frame:

```text
pixel (u, v) + depth z  →  3D point (x, y, z)
```

Then ROS TF / odometry / VSLAM transforms that point from the camera frame into a shared global/local frame such as `odom` or `map`:

```text
camera_0_depth_frame → base_link → odom
camera_1_depth_frame → base_link → odom
camera_2_depth_frame → base_link → odom
```

nvBlox then fuses all those observations into a **voxel grid**. A voxel is a small 3D cube, commonly around **5 cm × 5 cm × 5 cm** by default in Isaac ROS nvBlox. The `voxel_size` parameter default is `0.050` meters. ([nvidia-isaac-ros.github.io][2])

The reason the visualization looks like “cuboid reconstruction” is that objects are reconstructed as many small occupied/free 3D cells, or as a mesh extracted from those cells. So a box, pallet, shelf, or chair appears as a 3D cuboid-like structure.

### Why this avoids obstacles missed by planar LiDAR

A planar LiDAR might only see this:

```text
Top view / one scan height:

--------- laser plane ---------

Obstacle above the plane?  missed.
Obstacle below the plane?  missed.
Glass / thin object?       maybe missed.
Tabletop overhang?         often missed.
```

A depth camera or 3D LiDAR sees a **volume**, not only one slice:

```text
3D perception:

height ↑
       ███  shelf / box
       ███
       ███
       ───  2D LiDAR scan plane
       ███
       ███
```

So nvBlox can mark obstacles at different heights. Then it can generate either a full 3D ESDF for collision checking or a 2D costmap slice/projection for Nav2. nvBlox builds a TSDF voxel map for reconstruction and an ESDF distance field useful for path planning and collision checking. ([nvidia-isaac-ros.github.io][3])

### ROS topics involved

For multi-camera input, `nvblox_node` subscribes to up to four depth/color camera streams using topics like:

```text
camera_0/depth/image
camera_0/depth/camera_info
camera_0/color/image
camera_0/color/camera_info

camera_1/depth/image
camera_1/depth/camera_info
...
```

The same node can also subscribe to a `pointcloud` topic for 3D LiDAR input. The docs specify `PointCloud2` input for 3D LiDAR, plus pose/transform topics or TF for odometry. ([nvidia-isaac-ros.github.io][4])

The important outputs are:

```text
/nvblox_node/mesh
/nvblox_node/occupancy_layer
/nvblox_node/static_esdf_pointcloud
/nvblox_node/static_map_slice
/nvblox_node/combined_map_slice
```

The `mesh` is what you see in RViz. The `static_map_slice` or `combined_map_slice` is what navigation can consume as a costmap-like obstacle representation. ([nvidia-isaac-ros.github.io][4])

### Minimal real-world setup

For a drone or mobile robot, you would need:

```text
Sensors:
- Stereo/RGB-D camera, e.g. RealSense D455, ZED, Isaac Hawk, etc.
- Optional 3D LiDAR
- IMU
- Odometry source: VSLAM, visual-inertial odometry, Fast-LIO, PX4 odometry, etc.

Compute:
- Jetson Orin / AGX Orin / NVIDIA GPU machine

ROS 2 nodes:
- Camera driver
- Visual SLAM or odometry node
- TF tree broadcaster
- isaac_ros_nvblox
- RViz / Foxglove visualization
- Nav2 or MoveIt/cuMotion planner
```

For drones specifically, the key requirement is **accurate pose while moving**. nvBlox is not the localization system by itself. It needs the robot pose for each depth or point-cloud measurement. For aerial robots, that pose can come from PX4 odometry, visual-inertial SLAM, VIO, or LiDAR-inertial odometry. The nvBlox paper also describes drone usage with a 64-beam Ouster LiDAR and Fast-LIO pose estimation, integrating scans into a 5 cm resolution map. ([arXiv][5])

### In simple terms

nvBlox does this:

```text
“Every frame, tell me what the camera/LiDAR sees and where the robot was.
I will place that observation into a shared 3D grid.
Over time, I build a live 3D map.
From that map, I generate obstacles for the planner and a mesh for RViz.”
```

So the simultaneous visualization is possible because the system continuously receives sensor frames, transforms them into a common coordinate frame, fuses them into a GPU voxel map, extracts a mesh/costmap, and publishes updates to RViz while the robot is moving.

## Project implementation check - 2026-06-01

Short answer: partially applied.

The project has many parts of the 3D voxel-map idea, but it is not yet a complete live nvBlox-to-browser 3D voxel map.

What is applied in the backend:

- ROS 2 warehouse mapping profile exists for Isaac ROS nvBlox stereo.
- nvBlox topics are configured: mesh, occupancy layer, ESDF point cloud, static map slice, combined map slice, and back-projected depth.
- Bridge health checks know when nvBlox topics are present and expose readiness/missing-topic state.
- Warehouse flight readiness can require nvBlox before autonomy.
- Mapping stack lifecycle exists for starting/stopping warehouse mapping.
- Capture persistence can export nvBlox artifacts on stop through `/nvblox_node/save_ply` and `/nvblox_node/save_map`.
- Stored scan processing can register mesh/point-cloud/rosbag/quality assets.
- Exploration can consume an occupancy-grid snapshot from the ROS bridge and plan frontiers from that 2D grid.
- Backend live-map API exists: snapshot endpoint, update-publish endpoint, WebSocket stream, chunk schema, pose, scan path, health flags.

What is missing or only stubbed in the backend:

- No real ROS-side publisher is visible that subscribes to nvBlox mesh/ESDF/occupancy topics, converts them into `WarehouseLiveVoxelChunk` updates, and posts them to `/warehouse/live-map/{flight_id}/updates`.
- Live chunks store metadata and optional URLs, not actual compact voxel/mesh geometry payloads.
- Backend keeps live updates in process memory only. Restart loses the live map stream state.
- There is no spatial chunk store keyed by voxel block, bbox, LOD, sequence, and revision.
- No binary streaming path exists for high-rate map deltas. Current API is JSON-first.
- No direct browser-ready conversion from live nvBlox mesh blocks to glTF/3D Tiles chunks is implemented.
- Frontier planning uses 2D occupancy-grid snapshots, not a full 3D ESDF collision field.

What is applied in the frontend:

- Warehouse page includes a "Live Voxel Map" viewer.
- Frontend connects to `/warehouse/live-map/{flight_id}/stream` by WebSocket and loads snapshots over HTTP.
- Hook merges changed/removed chunk metadata and keeps bounded chunks/path history.
- UI shows stream state, coverage, drift, costmap freshness, mesh/point-cloud availability, and layer toggles.
- Stored scans can show mesh/point-cloud/path layers after capture processing.

What is missing or only stubbed in the frontend:

- Live viewer is a 2D canvas placeholder. It draws synthetic squares from chunk order, not actual mesh/point-cloud/voxel geometry.
- No Three.js/Cesium live 3D renderer for nvBlox mesh, ESDF, occupancy, or costmap slices.
- Chunk `url`, `bbox_local_m`, `block_ids`, and `point_count` are not used to fetch/render real geometry.
- No LOD, frustum culling, instancing, worker decode, or binary geometry decode path.
- Viewer controls like follow drone/reset camera are mostly visual shell.

What should be done to fulfill the idea:

1. Add a ROS live-map publisher node.
   - Subscribe to `/nvblox_node/mesh`, `/nvblox_node/static_esdf_pointcloud`, `/nvblox_node/occupancy_layer`, `/nvblox_node/static_map_slice`, `/nvblox_node/combined_map_slice`, and odometry/TF.
   - Convert nvBlox updates into stable spatial chunks with `id`, `kind`, `sequence`, `bbox_local_m`, `block_ids`, `point_count`, and pose.
   - Publish deltas to backend live-map API or directly to a backend ingest WebSocket.

2. Add a real chunk/artifact format.
   - Mesh chunks: glTF/GLB, meshopt-compressed glTF, or 3D Tiles.
   - Point-cloud chunks: LAS/LAZ/COPC, Potree-style octree, or binary packed XYZRGB.
   - Occupancy/ESDF chunks: compact binary voxel blocks with resolution, origin, dimensions, and typed arrays.
   - Keep JSON only as metadata/control plane.

3. Add backend spatial chunk service.
   - Persist latest chunk state per flight/session.
   - Store chunk files under session artifacts or object storage.
   - Track chunk lifecycle: added, updated, removed, finalized.
   - Serve range/cache-friendly chunk URLs.
   - Keep in-memory WebSocket fanout, but back it with Redis/Postgres/object storage for restart and multi-worker support.

4. Replace frontend placeholder with real 3D view.
   - Use Three.js for live local 3D scene.
   - Render mesh chunks as GLB/BufferGeometry.
   - Render point clouds with `Points`, typed arrays, and point budget.
   - Render occupancy/ESDF as instanced boxes, sparse voxel points, or slice planes.
   - Use drone pose, scan path, warehouse footprint, and chunk bbox in one coordinate frame.

5. Connect planning to 3D map output.
   - Feed ESDF/costmap age and obstacle clearance into flight readiness.
   - Use 3D ESDF for local collision checks around drone body.
   - Keep 2D projected occupancy for frontier exploration when needed, but preserve 3D height constraints for shelves, overhangs, and suspended obstacles.

What I would improve for content:

- Split this file into three sections: concept, current project status, implementation roadmap.
- Add exact repo references for each implemented piece: ROS config, backend services, frontend hook/viewer.
- Add a minimal data contract example for one live map update.
- Add a diagram showing real target architecture:

```text
nvBlox topics
  -> ROS live-map publisher
  -> backend chunk ingest + storage
  -> WebSocket metadata deltas
  -> browser fetches binary chunks
  -> Three.js renders live 3D map
```

- Define required sensor/pose assumptions clearly: RGB-D/stereo or 3D LiDAR, calibrated intrinsics/extrinsics, TF, odometry/VIO/LIO, GPU nvBlox runtime.
- Separate "live map" from "stored scan artifact"; they have different formats, latency, and quality goals.

What I would improve for speed:

- Do not stream full meshes over JSON. Stream small JSON deltas plus binary chunk URLs.
- Throttle ROS ingest to a target rate, e.g. 2-5 Hz mesh/point-cloud visual updates, while planner can consume higher-rate costmap/ESDF locally in ROS.
- Use spatial chunking with stable IDs so unchanged blocks are never resent.
- Add bbox/frustum culling in frontend before fetching/rendering chunks.
- Use Web Workers for binary decode and geometry preparation.
- Use meshopt/Draco for mesh compression where latency is acceptable.
- Use instanced rendering for occupancy voxels and GPU point rendering for point clouds.
- Keep a point/triangle budget and degrade gracefully by distance/priority.
- Cache chunk files with content hashes and immutable cache headers.
- Batch WebSocket updates per animation frame instead of applying every message immediately.
- Move heavy post-processing off request path into background jobs.
- Use Redis pub/sub or a queue for multi-process WebSocket fanout.
- Keep nvBlox and planning on the Jetson/ROS side; send browser only visualization-grade geometry.

Best next implementation step:

Build the ROS live-map publisher first. It is the missing bridge between real nvBlox output and the existing backend/frontend live-map shell. After that, replace the canvas placeholder with a Three.js renderer that consumes real chunk URLs and bbox metadata.
