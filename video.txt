VIDEO ANALYSIS PRODUCT AND IMPLEMENTATION RECOMMENDATIONS
Prepared: 2026-05-26
Scope: Drone App Video Analysis page, supporting agriculture, patrol/security,
photogrammetry, livestock/animal farm, infrastructure inspection, and emergency search.

===============================================================================
1. EXECUTIVE DECISIONS
===============================================================================

1. Do not route every task through one YOLO26 detector.
   YOLO26 is appropriate for visible object, instance-mask, oriented-box, pose, and
   tracked-event inference. It is not a substitute for calibrated vegetation indices,
   thermal/radiometric analysis, photogrammetric reconstruction, terrain products, or
   deterministic geofence/business rules.

2. Evolve "Video Analysis" into a task-profile driven Visual Analysis workspace.
   Video remains a valid source for patrol, livestock review, search, and inspections.
   Agriculture and photogrammetry need image sets, orthomosaics, multispectral bands,
   thermal rasters, RTK/GCP metadata, and multi-date comparisons in the same workflow.

3. Use YOLO26 where geometry creates operational value:
   - Detection: people, vehicles, cattle, wildlife, crop plants, equipment, defects.
   - Segmentation: weed/crop patches, water pooling, damaged surfaces, solar hotspots
     after thermal thresholding, animal outlines.
   - OBB: aerial vehicles, roofs, panels, linear assets, oriented crop plots.
   - Pose: human fall/posture triage only after domain validation; not safety verdict.
   - Tracking: patrol intrusions, animal counts, repeated target suppression.

4. Use task-specific models, calibrated sensors, and human review before action.
   No spray prescription, security escalation, welfare alert, or maintenance work order
   should be auto-published from an unvalidated generic model.

5. Resolve YOLO26 licensing before production commercialization.
   Ultralytics documents YOLO26 release on 2026-01-14 and lists AGPL-3.0 or
   Enterprise licensing. Their license page states proprietary/internal/production
   deployments using Ultralytics code or models require Enterprise licensing unless
   the wider application follows AGPL-3.0 obligations.

2. CURRENT APPLICATION BASELINE OBSERVED IN THIS REPOSITORY
===============================================================================

Existing useful foundation:
- Backend module `backend/modules/video_analysis/` handles recorded-video upload,
  queued offline inference, frame sampling, YOLO26 invocation, telemetry matching,
  crop evidence storage, job status, and detections.
- Accepted built-in models currently are `yolo26n.pt`, `yolo26s.pt`,
  `yolo26n-seg.pt`, and `yolo26s-seg.pt`; custom local `.pt` paths are allowed.
- Frontend `frontend/src/modules/video-analysis/` already presents upload controls,
  model/confidence/interval controls, video overlays, timeline, map, live detection
  log, and detection table.
- Adjacent modules already provide important domain building blocks:
  irrigation analytics, livestock geofence/isolation rules, patrol live vision,
  mapping/WebODM processing, missions, telemetry, alerts, and deliverables.

Gaps to close:
- A detection row is the main result; there is no first-class analysis profile,
  mission objective, raster layer, zone/polygon result, task metric, recommendation,
  review/approval state, export package, or model validation evidence.
- Current generic COCO YOLO26 models do not reliably represent aerial weeds, crop
  stress, livestock, small overhead targets, defects, or thermal anomalies.
- Segmentation weights can be selected, but the current detector output contract stores
  boxes rather than mask/raster/vector analysis products.
- Agriculture source flow is video-oriented, while crop/irrigation products normally
  require georeferenced still imagery/orthomosaics and often multispectral calibration.
- Photogrammetry is already a separate processing workflow; Video Analysis should link
  to its outputs rather than duplicate reconstruction with YOLO.

3. VIDEO / VISUAL ANALYSIS PAGE: FEATURES TO ADD
===============================================================================

A. Task and source selection
- Start with task profile: Patrol, Crop Scouting, Weed Mapping, Irrigation Stress,
  Stand Count, Livestock Count/Welfare Check, Photogrammetry QA, Asset Inspection,
  Search and Rescue.
- Source types: recorded RGB video, live stream, RGB image set, multispectral image
  set, thermal image set/video, existing orthomosaic/DSM, completed mapping job.
- Sensor metadata panel: camera, spectrum, radiometric calibration status, RTK/GPS,
  ground sampling distance (GSD), mission altitude, capture time, weather notes.
- Block invalid combinations: e.g. NDVI cannot run on RGB video alone; mapping quality
  cannot be guaranteed without overlap/georeferencing information.

B. Analysis profile builder
- Presets per task with allowed source, required model pipeline, confidence threshold,
  frame/tile sampling, minimum GSD, output type, severity rules, and export type.
- Model registry display: version, classes, sensor/input channels, validation dataset,
  per-class precision/recall, approval state, license, runtime target, last deployment.
- Choose edge preview vs server high-quality run. Show estimated processing time and
  result suitability: "triage", "review required", or "approved decision support".

C. Review canvas and evidence
- Keep existing overlay player/timeline/map, then add:
  mask and polygon layers; heatmaps; orthomosaic/DSM/thermal/vegetation-index layers;
  target tracks; compare-to-prior-flight swipe/difference view; uncertainty layer.
- Each finding must open evidence: source frame/tile, model/version, confidence,
  coordinates, sensor metadata, related track, reviewer status, and audit history.
- Enable correction: accept, reject, relabel, adjust polygon, merge/split tracks.
  Store corrected annotations for retraining candidates.

D. Domain result summaries
- Agriculture: affected hectares/percent, weed polygons, crop gaps, emergence count,
  canopy/stress zones, suspected irrigation issue zones, scouting waypoints.
- Patrol: unique people/vehicles, restricted-zone entry events, loitering duration,
  route/time/location, incident clips, acknowledged/escalated status.
- Livestock: unique count, distribution map, missing/isolated candidates, boundary
  exceptions, water/trough inspection flags, revisit route.
- Photogrammetry: coverage/overlap QA, blur and missing-area warnings, reconstruction
  assets, change-detection/annotation layers on orthomosaic or 3D model.
- Inspection: defect class, thermal hotspot, size/severity estimate, asset location,
  repeat-vs-new finding and work-order export.

E. Actions and exports
- Create targeted re-flight/scouting mission from zones or low-confidence findings.
- Export GeoJSON/Shapefile/GeoTIFF/CSV/PDF evidence report; agriculture additionally
  export prescription-map candidates only after agronomist approval.
- Send reviewed findings to Alerts, Deliverables, Mapping, or a work-order/webhook
  integration. Preserve source/result/model provenance.

F. Operations and safety
- Job stages: upload/ingest, quality check, processing, post-processing, awaiting
  review, approved/exported, failed/retryable.
- Quality flags: telemetry missing, GPS mismatch, low GSD, motion blur, low overlap,
  calibration missing, cloud/shadow interference, unvalidated model-domain pairing.
- Role-based approvals and tenant isolation for evidence/export.
- Metrics dashboard: latency, failed jobs, false-positive corrections, model drift by
  task/site/season, reviewer turnaround, hectares/minute or video minutes/minute.

4. HOW TO CONNECT TASKS WITH YOLO26, AND WHY
===============================================================================

Shared integration pattern:
1. A task profile chooses sources and a pipeline, not only a weight filename.
2. A common inference adapter executes YOLO26 detect/segment/OBB/pose or tracking
   when that task needs visual target geometry.
3. Domain post-processors convert geometry into business outputs: zone polygons,
   unique counts, incidents, coverage issues, inspection findings, or scout points.
4. Every output stores source asset, coordinates, timestamp, pipeline/model version,
   calibration/quality status, confidence, reviewer action, and export history.

Task-to-YOLO26 allocation:

Task: Patrol/security
- YOLO26 role: fine-tuned aerial detect model for person/vehicle/animal; tracking for
  unique intruders and loitering; optional pose model for review-only fall candidate.
- Why: events depend on visible entities over time and location. Generic frame boxes
  alone inflate event counts; tracking and restricted-zone logic create useful alerts.
- Non-YOLO parts: geofence/zone rules, telemetry, alert workflow, clip retention,
  thermal person-search model if thermal imagery is used.

Task: Crop stand count and gaps
- YOLO26 role: custom detect or instance-seg model for individual early-stage plants;
  tracking/deduplication if using video, preferably orthomosaic tile inference.
- Why: stand decisions depend on count, row spacing, skipped/double plants and gaps.
- Non-YOLO parts: georeferenced mosaic, crop row reconstruction, population/hectare,
  field boundary clipping, agronomic thresholds and reviewer approval.

Task: Weed mapping / spot spray candidates
- YOLO26 role: custom `-seg` model for weed instances/patches from suitable RGB
  imagery when plant-level boundaries are distinguishable.
- Why: masks can become treated-area polygons and quantify weed coverage better than
  boxes.
- Non-YOLO parts: multispectral crop/weed semantic model is often better where crop
  and weed appearance overlaps; convert reviewed weed polygons into prescription maps.

Task: Irrigation and crop stress
- YOLO26 role: optional visual detection of broken emitters, visible standing water,
  damaged pipes, or symptomatic plant clusters after zoning.
- Why not primary: poor irrigation is commonly a spatial spectral/thermal anomaly,
  not an object class visible reliably in ordinary RGB video.
- Primary method: calibrated multispectral/thermal orthomosaic; NDVI/NDRE/GNDVI or
  thermal anomaly zones; compare against prior flights and ground moisture/flow data.

Task: Livestock / animal farm
- YOLO26 role: custom aerial animal detect/segment model and tracker for count,
  position, distribution, movement, isolated-animal candidate and boundary exception.
- Why: animals are observable targets and movement history is operationally useful.
- Non-YOLO parts: existing collar/geofence rules, humane flight constraints, water/
  fence inspections, and veterinary review; video cannot diagnose disease reliably.

Task: Photogrammetry/survey
- YOLO26 role: not reconstruction core. Use after reconstruction for feature/asset
  annotation, obstacle/vehicle masks, panel/roof OBBs, defect candidate layers.
- Why: reconstruction requires feature matching, camera geometry, RTK/GCP, overlap,
  point clouds and orthorectification. Existing WebODM integration is correct core.
- Non-YOLO parts: WebODM/processing engine outputs orthomosaic, point cloud, DSM/DTM,
  textured model and measurement products.

Task: Infrastructure/solar/roof inspection
- YOLO26 role: segment or OBB model for visible defect/component localization and
  asset-level repeated finding association.
- Why: locating findings on repeat captures supports maintenance triage.
- Non-YOLO parts: radiometric thermal threshold/calibration, digital twin alignment,
  severity rules and inspector sign-off.

Task: Search and rescue / wildlife survey
- YOLO26 role: RGB/thermal domain-specific detection plus tracking for candidate
  people or animals and duplicate suppression.
- Why: operator needs georeferenced candidate targets quickly.
- Non-YOLO parts: thermal sensor pipeline, flight safety, priority routing and strict
  human confirmation before response decisions.

Recommended YOLO26 product integration:
- Keep official `yolo26n/s` only for demonstrations and broad preliminary triage.
- Add profile-bound custom weights and validated output schemas:
  `storage/ml_models/patrol/aerial_person_vehicle_yolo26s.pt`
  `storage/ml_models/crops/stand_count_<crop>_yolo26s-seg.pt`
  `storage/ml_models/crops/weed_<crop>_yolo26s-seg.pt`
  `storage/ml_models/livestock/cattle_yolo26s-seg.pt`
  `storage/ml_models/inspection/<asset>_defects_yolo26s-seg.pt`
- Add OBB options (`yolo26*-obb.pt`) for overhead rotated objects and custom training.
- For small aerial targets, evaluate YOLO26 P2 architecture configuration; official
  docs state P2/P6 are architecture YAML variants requiring training, not released
  pretrained weights.
- Run tracking using YOLO-supported video tracking after detection; persist track/event
  summaries separately from raw detections.

5. INDUSTRY WORKFLOWS AND RESULTS, PLUS APPLICATION RECOMMENDATIONS
===============================================================================

The company examples below describe publicly documented product workflows/results.
They identify market expectations; they do not imply access to proprietary models.

AGRICULTURE: CROP HEALTH, WEEDS, STAND COUNT, APPLICATION PLANNING

Company/product: DJI Agriculture Mavic 3 Multispectral + SmartFarm
Public workflow/result:
- Captures RGB plus green/red/red-edge/NIR data with RTK positioning.
- SmartFarm generates NDVI data for localized crop treatment; DJI describes automatic
  scouting for crop abnormalities.
Recommendation for this app:
- Add multispectral image-set ingest, calibration metadata, NDVI/NDRE/GNDVI raster
  layers, zone polygons, scout-waypoint generation and treatment-review workflow.
- Do not label RGB YOLO detections as irrigation/crop-health diagnosis.

Company/product: PIX4Dfields
Public workflow/result:
- Processes drone/satellite imagery offline at field edge; produces crop-health maps,
  vegetation indices, weed identification and customizable prescription maps for
  targeted/variable-rate application.
Recommendation for this app:
- Make zonation and reviewed GeoJSON/GeoTIFF prescription export core agriculture
  results. Preserve machinery format integration as a later controlled export.

Company/product: Sentera FieldAgent / SmartScripts Weeds
Public workflow/result:
- Planned imagery is analyzed for stand count, canopy cover and crop health; produces
  mosaics such as NDVI/NDRE/VARI; SmartScripts Weeds supplies weed maps and spray
  prescriptions.
Recommendation for this app:
- Build reusable analysis recipes by crop/growth stage and include stand uniformity,
  canopy coverage, weed-pressure map and prescription approval in agriculture profile.

Company/product: DroneDeploy Agriculture
Public workflow/result:
- Provides field-edge/offline stand count for early corn/soybean emergence; map layers
  include Plant Health and Elevation; crop-health results can guide prescription work.
Recommendation for this app:
- Add stand-count density/gap reports, early-emergence task preset and offline-ready
  review experience. Link elevations and crop-health zones rather than showing only
  bounding boxes.

PATROL, SECURITY AND REMOTE SITE MONITORING

Company/product: Percepto AIM
Public workflow/result:
- Manages remote autonomous drone operations; organizes data on geospatial map;
  applies automated industry analytics to visual data and reports anomalies/trends.
  Listed uses include visual/thermal inspection and security/emergency response.
Recommendation for this app:
- Combine live and offline patrol runs into incident workflow: detect -> track ->
  apply zone/time rule -> save evidence clip -> reviewer triage -> alert/report.
- Add repeatable routes, compare-by-location findings, thermal source type and trend
  reports. Current raw detection log is necessary but not incident management.

PHOTOGRAMMETRY, SURVEY AND ASSET DIGITAL TWINS

Company/product: Skydio 3D Scan
Public workflow/result:
- Autonomous, asset-aware capture manages coverage/overlap around structures and
  terrain; exports high-resolution imagery to photogrammetry workflow for 2D maps
  and inspection-grade 3D models/digital twins.
Recommendation for this app:
- Keep capture/reconstruction under Mapping; let Visual Analysis select completed
  orthomosaics/models and create annotated findings or change comparisons.

Product used in this repository: WebODM
Public workflow/result:
- Generates georeferenced orthomosaics, point clouds, DSM/DTM elevation models and
  textured 3D models from aerial imagery, with offline operation supported.
Recommendation for this app:
- Surface WebODM asset links and QA status in Visual Analysis; use reconstructed
  coordinate products for geospatial finding layers and repeat-inspection comparison.

ANIMAL FARM / LIVESTOCK

Company/product: CattleQuants
Public workflow/result:
- Collects aerial images, applies AI counting, and returns shareable inventory report;

Recommendation for this app:
- Add aerial count mission, confidence/duplicate review, count-by-paddock report and

Company/product: GrazeMate
Public workflow/result:
- Describes scheduled missions locating cattle and checking infrastructure; publishes
  AI trough/fence detection and pasture-monitoring functions.
Recommendation for this app:
- Enrich existing livestock geofence/isolation rules with drone-derived animal
  locations, trough/fence inspection candidates, distribution maps and follow-up
  waypoints; keep animal movement/stress safeguards explicit.

INSPECTION

Company/product: Flyability Inspector / Elios 3
Public workflow/result:
- Inspection software associates points of interest with corresponding video frames
  to analyze defects and collaborate on captured inspection data.
Recommendation for this app:
- Use evidence-first inspection findings: asset coordinate, frame, annotation,
  severity, reviewer, prior finding comparison and work-order/export status.

6. TASK-SPECIFIC MODEL AND ANALYTICS CANDIDATES
===============================================================================

General rule:
- "Available pretrained" does not mean fit for production. Validate with local sensor,
  altitude, season, geography, lighting and target-size data before enabling decisions.
- Store benchmark reports with each deployed model: classes, data lineage, holdout
  split by site/date, metrics, failure modes, approved task/source, and license.

Task | Candidate model/analytics | Input | Result | Recommendation
-----|---------------------------|-------|--------|---------------
General demo detection | Official `yolo26n.pt` / `yolo26s.pt` | RGB frames | COCO boxes | Retain for demo/triage only; not specialist conclusion.
General masks | Official `yolo26n-seg.pt` / `yolo26s-seg.pt` | RGB frames | COCO instance masks | Implement mask storage/rendering, but train specialist weights.
Aerial patrol: person/vehicle | YOLO26 detect fine-tuned on VisDrone plus local patrol captures; test P2 configuration for small targets | RGB aerial video | boxes/tracks/incidents | First custom production candidate; integrate existing zone/anomaly pipeline.
Patrol thermal person search | Thermal-domain detector trained/validated on actual thermal sensor data; optionally YOLO26-based after dataset/license approval | Thermal frames | candidates/tracks | Separate from RGB model; always operator confirm.
Crop stand count | YOLO26 detect or `-seg` fine-tuned per crop and growth stage using app captures/annotations | High-GSD RGB orthomosaic tiles | plants, gaps, density/ha | Prioritize after agriculture raster ingest; never mix crops/stages blindly.
Weed patches (RGB) | Custom YOLO26-seg trained per crop/site/stage | RGB tiles | masks/polygons/coverage | Good actionable YOLO path where weed geometry visible.
Weed patches (multispectral) | WeedMap semantic segmentation reference; WeedsGalore dataset/reference models for maize; retrain/evaluate for local crop | Calibrated multispectral orthomosaic tiles | crop/weed/background raster | Use dedicated multispectral service; YOLO RGB is not equivalent.
Poor irrigation/crop stress | Vegetation-index and thermal anomaly pipeline with temporal baseline; optional ML segmenter after ground truth | Calibrated multispectral/thermal map plus soil/flow checks | stress zones, inspection points | Do not search for "poor irrigation YOLO" shortcut; apparent stress has multiple causes.
Livestock count | YOLO26 detect/seg adapted using local overhead data; CattleEyeView research dataset is relevant for detection, pose, tracking and segmentation experiments | RGB top-down video/images; thermal separately | animal tracks/count/map | Couple with existing deterministic geofence/isolation logic.
Livestock welfare | Pose/mobility research only; vet-reviewed rule set | Close enough validated imagery + collar data | revisit candidate | Do not claim diagnosis from drone video.
Photogrammetry | Existing WebODM processing | Geotagged overlapping still images | ortho/point cloud/DSM/DTM/3D | No YOLO replacement. Link outputs into analysis page.
Asset/roof/solar inspection | Custom YOLO26-seg/OBB for visible findings plus calibrated thermal anomaly processor | repeat RGB/thermal captures mapped to asset | defect/hotspot polygon | Require inspector review and baseline comparison.
Wildlife survey | species-specific detector trained on local aerial imagery; track/deduplicate | RGB/thermal aerial captures | candidate count/map | Treat count as reviewed estimate, respecting wildlife flight constraints.

Public data/reference candidates worth evaluating:
- VisDrone: drone-view images/video annotations for detection/tracking/crowd-count
  tasks; Ultralytics documents YOLO26 training using `VisDrone.yaml`. Suitable starting
  point for patrol overhead targets, then fine-tune on local site data.
- WeedMap: publicly released UAV multispectral crop/weed/background semantic
  segmentation framework and sugar-beet dataset. Useful for model/data design and
  weed-raster baseline, not a universal crop model.
- WeedsGalore: multispectral, multitemporal UAV maize crop/weed semantic and instance
  segmentation dataset with reference code. Useful agriculture evaluation source.
- CattleEyeView: top-down cattle video research dataset spanning counting, detection,
  pose, tracking and instance segmentation. Useful evaluation/pretraining research
  source; local aerial validation still required.

Model metrics required before enabling a task:
- Object/instance tasks: per-class precision, recall, mAP, false positives per hectare
  or per video-minute; count error; track ID switches where tracking used.
- Zone/segmentation tasks: IoU/Dice, affected-area error, polygon boundary tolerance,
  prescription-area false-positive cost.
- Geospatial tasks: localization error against surveyed/RTK truth.
- Change/trend tasks: repeatability across same-location missions and false change rate.
- Operational tests: runtime, GPU/edge memory, queue throughput, failure recovery,
  missing telemetry behavior, audit/export correctness.

7. ENRICHED ANALYSIS FLOWS BY TASK
===============================================================================

7.1 Crop scouting and weed management
Input:
- Field boundary, crop/type/stage, RGB and preferably multispectral image set,
  calibration panel/sunlight metadata, RTK/GPS, prior survey.
Flow:
1. Validate overlap, GSD, bands, calibration and field coverage.
2. Build orthomosaic; calculate approved vegetation indices.
3. Tile inference for crop/weed segmentation and/or stand count.
4. Vectorize masks into weed/stress/gap zones and quantify hectares/percentage.
5. Compare prior mission and rank scout points by severity/uncertainty.
6. Agronomist accepts/edits findings and generates prescription candidate/export.
Results/UI:
- Index layer selector, weed polygons, population density/gaps, change map,
  scout route, report, reviewed prescription export.

7.2 Irrigation assessment
Input:
- Multispectral/thermal imagery, field zones/irrigation layout if available, prior
  baseline, optional moisture/flow data.
Flow:
1. Validate radiometric/calibration and capture-time consistency.
2. Construct raster and vegetation/thermal layers.
3. Find anomalous zones relative to local rows and historical baseline.
4. Correlate with irrigation lines, terrain and sensor readings.
5. Generate "suspected under-irrigated/overwatered/uneven" inspection targets.
6. Human verifies; only verified result becomes maintenance/treatment task.
Results/UI:
- Stress/thermal map, anomaly polygons with evidence and alternative causes,
  water-line overlay, follow-up mission/ground check.

7.3 Patrol and security
Input:
- Live or recorded RGB/thermal stream, route, restricted polygons, time policy,
  telemetry, prior events.
Flow:
1. Detect and track person/vehicle/animal with aerial-domain model.
2. Project detections to map where telemetry quality is sufficient.
3. Run zone/loitering/direction/time rules; group frames into incidents.
4. Save short evidence clip/keyframes; suppress duplicate alerts by track/incident.
5. Operator triages and escalates/archives.
Results/UI:
- Live incident queue, track timeline, map trail, restricted-zone events,
  acknowledged/escalated state and evidence export.

7.4 Livestock / animal farm
Input:
- Scheduled RGB/thermal survey, paddock geofences, animal registry/collar positions,
  trough/fence asset layers, humane altitude/no-chase policy.
Flow:
1. Detect/track animals and calculate reviewed count per paddock.
2. Fuse with collar positions without assuming visual identity.
3. Apply boundary/isolation rules; identify unresolved count mismatch.
4. Run infrastructure inspection candidate model on troughs/fences if validated.
5. Plan low-disturbance follow-up waypoint mission for flagged areas.
Results/UI:
- Reviewed herd count, distribution and density, isolated/boundary candidates,
  water/fence flags, missing-location workflow and audit report.

7.5 Photogrammetry and site survey
Input:
- Geotagged still images, RTK/GCP, mission overlap/coverage, selected WebODM job.
Flow:
1. Capture-quality QA: blur, coverage gaps, overlap and positioning warnings.
2. Process through existing mapping/WebODM pipeline.
3. Display orthomosaic, point cloud, DSM/DTM, textured model and quality metadata.
4. Optionally execute YOLO annotation/change models on orthomosaic or asset imagery.
5. Create measurement/annotation deliverable.
Results/UI:
- Reconstruction asset viewer/link, coverage re-flight needs, annotated features,
  comparison to earlier map/model, deliverable package.

7.6 Infrastructure inspection
Input:
- Asset registry, planned repeated capture positions, RGB/thermal imagery/video,
  prior accepted findings.
Flow:
1. Align capture to asset/digital twin.
2. Run visible defect segmentation/OBB model and thermal anomaly processor.
3. Compare to prior accepted findings; mark new/growing/resolved.
4. Inspector validates type/severity and produces action.
Results/UI:
- Asset finding list, frame-to-map/model evidence, thermal overlay, progression,
  work-order/export integration.

7.7 Search and rescue / wildlife
Input:
- Search polygon/transects, RGB/thermal stream, species/person model and mission rules.
Flow:
1. Detect candidates and track to suppress duplicate reports.
2. Rank by confidence, sensor agreement and last-seen coordinate.
3. Operator confirms target; create safe revisit route or response event.
Results/UI:
- Candidate queue/map, thermal/RGB evidence pair, coverage map, reviewed count/event.

8. RECOMMENDED CLEAN ARCHITECTURE FOR IMPLEMENTATION
===============================================================================

Preserve current module-first structure. Keep routes thin and run long processing in
workers. Suggested boundaries:

`backend/modules/visual_analysis/` (or expand/rename current `video_analysis`)
- `api.py`: upload/register sources, create runs, fetch results/reviews/exports.
- `schemas.py`: task profiles, sources, run config, finding/layer/review/export DTOs.
- `models.py`: analysis run, asset, finding, track, layer, review and model deployment.
- `repository.py`: tenant-scoped persistence only.
- `service/profile_service.py`: validate task/source/model compatibility.
- `service/run_service.py`: orchestrate run lifecycle/idempotency.
- `service/review_service.py`: reviewer edits/approval and audit events.
- `service/export_service.py`: approved export requests.

`backend/modules/visual_analysis/pipelines/`
- `yolo_pipeline.py`: YOLO adapter outputting boxes/masks/OBBs/tracks.
- `agriculture_pipeline.py`: raster indices, segmentation zones, stand/weed outputs.
- `irrigation_pipeline.py`: anomaly zoning plus sensor correlation.
- `patrol_pipeline.py`: track-to-incident/zone event conversion.
- `livestock_pipeline.py`: count/fusion/rule outputs.
- `inspection_pipeline.py`: asset finding/change outputs.
- Do not put photogrammetric reconstruction here; call existing Mapping service/assets.

`backend/modules/model_registry/`
- Model artifact, task, version, input sensor, class map, validation metrics, license,
  storage URI, deployment state and rollback history.
- Deployment gate rejects an unapproved model for actionable export profiles.

`backend/entrypoints/workers/`
- Idempotent analysis tasks with retry policy, progress checkpoints, cancellation,
  failure table/dead-letter behavior and structured safe logs.

Key persistence additions:
- `analysis_profiles`: profile code/version/task, allowed inputs and output contract.
- `analysis_assets`: video/image-set/raster/mapping-asset links and sensor metadata.
- `analysis_runs`: input hash, profile/model versions, status/progress/quality flags.
- `analysis_findings`: point/box/mask/polygon/raster references, confidence, severity.
- `analysis_tracks` and `analysis_events`: timed patrol/livestock semantics.
- `analysis_layers`: GeoTIFF/vector/index/change-map deliverables.
- `analysis_reviews`: accept/reject/edit/approve, user, timestamp, reason.
- `model_deployments` and `model_evaluations`: metrics, site/date split, approval.
- All organization-owned rows must include/enforce `org_id`; exports must require
  tenant authorization and approval status where actionable.

API additions:
- `GET /visual-analysis/profiles`
- `POST /visual-analysis/assets` for video/image-set/raster/mapping-job registration
- `POST /visual-analysis/runs`
- `GET /visual-analysis/runs/{id}` and `/findings`, `/layers`, `/events`
- `POST /visual-analysis/findings/{id}/review`
- `POST /visual-analysis/runs/{id}/exports`
- `GET /model-registry/deployments?task=...`

Frontend module changes:
- Add task-profile/source wizard before current controls.
- Replace fixed model select with approved models returned by profile.
- Keep player/timeline/table/map for video-compatible runs.
- Add geospatial layer canvas for orthomosaic/index/mask/polygon products.
- Add Findings Review panel, Results Summary cards, Quality panel and Exports panel.
- Render empty/loading/failed/awaiting-review/approved/exported states explicitly.
- Keyboard-operable layer controls, accessible finding table and non-color-only
  severity labels are required.

Security, reliability and governance:
- Virus/type/size validation and controlled object storage for uploaded media.
- Idempotency key based on tenant + source hash + profile/model/config version.
- Immutable source/result provenance and audit trail for review/export.
- Retention controls for sensitive patrol footage and people detections.
- Human approval on security escalation, livestock welfare conclusions, spray maps and
  maintenance action exports until defined validation and policy gates are passed.

Stage 0 - Product, compliance and validation contract
Goal:

  Review; choose whether Livestock Count follows next.

- Define tenant/privacy/retention requirements, review roles and export approvals.
- Define metric gates per task using section 6.
Done when:

  acceptance document exist.

Stage 1 - Unify existing source and run domain without breaking current video page
Goal:
- Convert current video-only job concept into extensible analysis runs.
Work:

- Add model registry records for current YOLO26 weights as "demo/triage", not approved
  domain models.
- Add idempotent worker lifecycle, progress events, audit fields and org scoping tests.
Done when:
- Existing offline video run still works; same system can register image/raster/
  mapping sources and refuses incompatible profile/input combinations.

Stage 2 - Page workflow and human review foundation
Goal:
- Make results inspectable and correctable before specialist models arrive.
Work:
- Build task/source selection, model suitability/quality panel, explicit run stages,
  finding evidence drawer and accept/reject/relabel/edit workflow.
- Extend result format/rendering for masks and polygons.
- Add reviewed findings export to GeoJSON/CSV/PDF with provenance.
Done when:
- Operator can run current detection, review evidence, correct results and export a
  traceable reviewed report with keyboard-accessible controls.

Stage 3 - Patrol production candidate
Goal:
- Deliver domain-validated incident workflow from strongest existing code path.
Work:
- Collect/label local aerial patrol footage; start with VisDrone-to-local YOLO26
  fine-tuning; test standard and small-target/P2 configurations.
- Integrate tracker, restricted-zone/loitering event aggregation, duplicate alert
  suppression and keyframe/clip evidence into shared findings/events contract.
- Connect reviewed event to Alerts/Deliverables.
Done when:
- Holdout validation passes defined event false-positive/recall/localization gates;
  operator manages incidents instead of isolated boxes.

Stage 4 - Mapping/raster and agriculture substrate
Goal:
- Accept correct sensor/source products for agronomic analysis.
Work:
- Link completed Mapping/WebODM assets; ingest multispectral images and calibration
  metadata; expose orthomosaic/raster/vector layers in frontend.
- Produce vegetation-index and temporal comparison layers with quality warnings.
- Reuse field polygons and mission planning for scouting/re-flight waypoint creation.
Done when:
- User can view field-index/change layers, select zones and generate a reviewed
  scouting mission without representing stress as a generic YOLO box.

Stage 5 - Weed and stand-count specialist workflows
Goal:
- Convert imagery into actionable crop geometry and reviewed recommendations.
Work:
- Annotate local crops/stages; evaluate WeedMap/WeedsGalore references and train
  fit-for-source segmentation; train YOLO26-seg where RGB instance geometry performs.
- Train stand-count model per selected crop/growth stage; output density and gap zones.
- Add weed coverage, count/gap summaries and reviewed prescription candidate export.
Done when:
- Site/date holdout metrics meet approved thresholds; agronomist can edit zones and
  approve exports; unsupported crop/stage combinations are blocked.

Stage 6 - Irrigation stress workflow
Goal:
- Provide defensible irrigation investigation rather than visual guess.
Work:
- Add thermal/multispectral anomaly processing, baseline comparison and irrigation
  asset overlay; fuse optional flow/soil measurements.
- Classify outputs as suspected causes with evidence and follow-up verification steps.
- Connect confirmed findings to maintenance/scouting work.
Done when:
- Repeated field trials correlate zones with verified ground findings and application
  shows uncertainty/alternative cause, calibration and verification status.

Stage 7 - Livestock and inspection extensions
Goal:
- Reuse analysis foundation for animal farm and physical assets.
Work:
- Evaluate cattle aerial model against local imagery and CattleEyeView-based research;
  combine reviewed count with existing geofence/isolation/collar rules.
- Add trough/fence or selected asset inspection models and repeat-finding comparison.
- Add humane-operation constraints and inspector/welfare-review approval policies.
Done when:
- Reviewed herd-count/distribution workflow and selected inspection workflow pass
  domain validation; no unvalidated health/maintenance conclusion is shown as fact.

Stage 8 - Production optimization and feedback loop
Goal:
- Operate at scale and improve models safely.
Work:
- Export approved models for required edge/server runtime after license confirmation;
  benchmark throughput/cost and support rollback.
- Monitor corrections, drift, failure rates, latency, geolocation quality and model
  metrics by source/site/season.
- Convert reviewer corrections into controlled annotation/retraining datasets with
  versioned evaluations and deployment approval.
Done when:
- Observability, rollback, model governance, privacy retention, disaster recovery and
  retraining release process are tested in production-like conditions.

Recommended implementation sequence:
1. Stages 0-2: platform correctness and review UX.
2. Stage 3: patrol, since existing live/offline vision and telemetry foundations exist.
3. Stage 4: raster/mapping substrate required for honest agriculture output.
4. Stages 5-6: agriculture specialist value.
5. Stage 7: livestock/inspection reuse.
6. Stage 8: scale and governed improvement.

11. FINAL PRODUCT POSITION
===============================================================================

Build one geospatial, reviewable Visual Analysis workspace, not one generic detector
screen. YOLO26 should provide fast visual geometry where validated; mapping engines,
spectral/thermal processors, rules and humans should supply the domain meaning.
This design fits existing modules, creates useful task outputs, and avoids turning
confidence-scored boxes into unsupported agricultural, security or welfare decisions.
