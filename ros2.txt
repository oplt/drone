# ROS 2 Warehouse Workspace Flow Report

## Executive Summary

The warehouse bridge flow is now separated across the repo:

* [x] `gazebo`: Gazebo sim, `WAREHOUSE_TOPIC_PROFILE=gazebo`, Gazebo launch/bridge, sim time, Gazebo video.
* [x] `isaac`: Isaac/NVBlox external stack, `WAREHOUSE_TOPIC_PROFILE=isaac_ros_nvblox_stereo`, Isaac contract/preflight, wall time.
* [x] `real_device`: physical drone/sensors, `WAREHOUSE_TOPIC_PROFILE=real_device`, hardware launch, wall time, optional real camera/IMU/lidar/VSLAM commands, no Gazebo transport.

The app/backend/helper nodes no longer depend on Isaac, Gazebo, or real-device topic names. They consume the stable warehouse contract:

* [x] `/warehouse/contract/rgb/image`
* [x] `/warehouse/contract/depth/image`
* [x] `/warehouse/contract/imu`
* [x] `/warehouse/contract/points`
* [x] `/warehouse/contract/odometry`
* [x] `/warehouse/contract/local_odometry`
* [x] `/warehouse/contract/map/*`

Source-specific topics live only in `warehouse_topics.yaml` `source_profiles` and are republished into the contract by `warehouse_topic_adapter`.

## Implemented Flow Boundary

* [x] Backend source of truth: `backend/modules/warehouse/service/bridge_flow.py`.
* [x] Backend env setting: `WAREHOUSE_BRIDGE_FLOW=gazebo|isaac|real_device`.
* [x] Backend perception status/start payload includes `bridge_flow`.
* [x] Backend bridge/mapping process managers choose launch/profile/env by flow.
* [x] Backend video and readiness checks use flow resolver so real device is not hijacked by stale Gazebo env.
* [x] ROS topic registry has `gazebo`, `isaac_ros_nvblox_stereo`, and `real_device`.
* [x] ROS topic registry exposes stable contract topics to all helper nodes.
* [x] ROS source profile mappings are isolated for adapters only.
* [x] ROS launch files exist for Gazebo, Isaac, and real device.
* [x] ROS launches start `warehouse_topic_adapter` before helper nodes.
* [x] Real-device flow has `defaults_real_device.yaml` and `real_device_warehouse_mapping.launch.py`.
* [x] Env examples document real-device driver command hooks.
