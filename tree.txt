DRONE BACKEND - RECOMMENDED MODULAR ARCHITECTURE
================================================

Scope
-----
Architecture recommendation after inspecting current backend package. This is a
target structure and migration guide, not an assertion that files already live
in these locations.

Current stack observed
----------------------
- FastAPI application: backend/api/api_main.py
- SQLAlchemy async persistence plus Alembic migrations
- PostgreSQL/PostGIS, Redis, MinIO/S3
- Celery workers and beat scheduler
- MAVLink/drone runtime, MQTT, WebSockets, OPC UA, ROS2 bridge
- WebODM/photogrammetry and patrol ML processing
- OpenTelemetry and Prometheus integration

Main architecture problems observed
-----------------------------------
- Mixed organization: technical-layer folders coexist with partial feature
  folders, making ownership unclear.
- API layer owns application and persistence logic. Routes directly import
  SQLAlchemy/models/session in fields, mapping, irrigation, auth, admin,
  integrations, analytics, templates, fleet, compliance, webhooks, and others.
- Central model file owns all domains: backend/db/models.py is 1786 lines.
- Major files exceed clean-module limits, especially:
  backend/api/routes/routes_flights.py              2467 lines
  backend/db/models.py                              1786 lines
  backend/flight/missions/private_patrol.py         1776 lines
  backend/drone/orchestrator.py                     1593 lines
  backend/flight/preflight_check/mission_specific.py 1493 lines
  backend/api/routes/routes_mapping.py               964 lines
  backend/api/routes/routes_warehouse.py             713 lines
  backend/services/alerts/engine.py                  653 lines
  backend/db/repository/telemetry_repo.py            587 lines
  backend/services/photogrammetry/service.py         561 lines
- Hardware adapters, mission use cases, realtime publishing, persistence, and
  long-running runtime control are coupled through global instances and imports.
- Worker task modules contain SQL/persistence behavior instead of invoking
  feature use cases.
- No Python test modules were found under backend/tests even though pytest is
  configured for that location.

Architecture decision
---------------------
Use a domain-first modular monolith under the existing backend package.

Reason:
- Product already has strongly distinct capabilities: mission execution,
  telemetry, mapping, irrigation, warehouse inspection, patrol/ML, livestock,
  alerts, identity, scheduling, exports, and external delivery.
- A layer-first layout has already produced giant routes/models and cross-domain
  imports.
- A modular monolith avoids service/network overhead while enforcing ownership
  boundaries; domains can be extracted later only when operational need exists.
- Retaining backend as package root avoids an unnecessary backend.app import
  migration while modules are extracted.

Dependency direction
--------------------
entrypoints/api or entrypoints/workers
  -> modules/<domain>/api or modules/<domain>/jobs
  -> modules/<domain>/application
  -> modules/<domain>/domain
  -> modules/<domain>/ports

infrastructure/adapters implement ports and may depend on framework/client SDKs.
core may be imported by all modules, but may not import feature modules.
One module uses another through its application facade or published events,
never by querying another module's ORM models directly.

Module rules
------------
- Each domain owns its models, repository, application service, API schemas,
  authorization rules, jobs, and tests.
- API files call services only. No SQLAlchemy statements, filesystem writes,
  Celery calls, MAVLink calls, WebODM calls, or event fan-out in API files.
- Services enforce tenant scope and invariants and invoke domain-owned
  repositories or explicit ports.
- Repositories contain persistence only and apply org/project scopes for
  tenant-owned entities.
- Models contain ORM mappings and constraints only.
- Jobs are thin worker adapters: deserialize task input, call service,
  record status, publish safe events.
- Infrastructure implements ports. It must not own product workflow.
- Realtime delivery uses domain events through publisher ports, rather than
  importing a global WebSocket manager throughout domains.
- Long-running mission and mapping execution records durable state first;
  worker/runtime processing must be idempotent and restart-safe.

File limits to enforce
----------------------
- API route/controller file: 220 effective lines maximum.
- Service/use-case file: 300 maximum.
- Repository/data-access file: 250 maximum.
- Schema/DTO/model file: 250 maximum.
- Worker/integration adapter file: 280 maximum.
- Any Python source file: 400 maximum.

Migration order
---------------
Phase 0 - Guardrails
- Add backend/tests skeleton, file-size checker, import-boundary checker, and
  stable structured error handler.
- Keep current HTTP paths and database schema unchanged.

Phase 1 - Split persistence ownership
- Move Base/session setup to core/database.
- Split db/models.py by module while keeping Alembic metadata aggregation.
- Move repository functions to domain repositories; require tenant scoping.

Phase 2 - Remove persistence from HTTP routes
- Start with routes_flights.py, routes_mapping.py, routes_warehouse.py,
  routes_field.py, and routes_irrigation.py because they have highest size and
  coupling risk.
- Preserve response contracts; introduce services behind existing endpoints.

Phase 3 - Isolate runtime and adapters
- Split Orchestrator into vehicle runtime coordination, execution, telemetry,
  and recovery services.
- Put MAVLink, maps, MQTT, WebSocket/Redis fan-out, video, object storage, and
  WebODM behind ports/adapters.

Phase 4 - Normalize async processing
- Convert Celery modules to thin task adapters calling module jobs/services.
- Add idempotency, retries/timeouts, status transitions, and outbox-backed
  notifications for mapping, deliverables, alerts, and webhooks.

Phase 5 - Tests and enforcement
- Add contract tests for existing routes, auth and tenant isolation.
- Add integration tests for PostgreSQL/PostGIS repositories and Celery jobs.
- Enforce Ruff, mypy, pytest, file-size, and boundary checks in CI.

Non-negotiable invariants
-------------------------
- User and organization scope is authorized server-side for every tenant-owned
  field, mission, asset, alert, export, webhook, fleet, and mapping record.
- Mission start requires valid state and applicable preflight decision.
- Mission command transitions are validated and audited.
- Telemetry/event ingestion remains idempotent by stable frame/event key.
- Mapping, irrigation, exports, webhooks, and alert delivery survive retries
  without duplicate externally visible effects.
- Credentials, JWTs, raw private imagery, and unrestricted prompts are never
  logged.
