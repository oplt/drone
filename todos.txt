## 1. Executive summary
- This repo already has the hard parts most drone MVPs never reach: a real mission orchestrator, mission-specific preflight, agriculture grid planning, telemetry/MQTT/video plumbing, alerts, and a credible photogrammetry pipeline. Preserve those foundations.
- The main weaknesses are in the operational hot path, not feature breadth: telemetry is ingested in multiple places, runtime state is partly in memory/module globals, video delivery is MJPEG-heavy, and the main API/UI surfaces are too monolithic for low-latency field use.
- The biggest opportunity is to turn the current mission/planning stack into a durable operations product: one telemetry bus, one persisted mission-runtime model, a premium live/replay workspace, and agriculture outputs that go beyond orthomosaic generation into action.

## 2. Speed and performance improvements
- [Quick win] Unified telemetry/event bus; problem: `backend/drone/orchestrator.py`, `backend/messaging/websocket.py`, and `backend/messaging/mqtt.py` each touch the live vehicle/event path differently; why it matters: duplicated MAVLink reads, inconsistent ordering, avoidable CPU/network load, harder backpressure; proposed solution: make `Orchestrator` the single ingest point and publish normalized telemetry/event envelopes to websocket, MQTT, alerts, DB, and ML consumers; expected impact: very high latency/reliability gain; implementation complexity: medium; likely code areas affected: `backend/drone/orchestrator.py`, `backend/messaging/websocket.py`, `backend/messaging/mqtt.py`.
- [Quick win] Batch telemetry and event persistence; problem: bulk insert support exists but the live path underuses it, while event writes are still too chatty; why it matters: DB commit pressure will become a bottleneck before flight logic does; proposed solution: bounded ingest queues, periodic bulk writes, a hot telemetry table plus colder raw-event storage, and precomputed minute summaries for dashboards/replay; expected impact: high throughput and lower DB cost; implementation complexity: medium; likely code areas affected: `backend/db/repository/telemetry_repo.py`, `backend/db/models.py`, `backend/api/routes/routes_analytics.py`.
- [Quick win] Persist mission/preflight runtime; problem: active preflight and mission runtime state lives in module globals and duplicated in-memory stores; why it matters: a process restart can lose operational state and break auditability; proposed solution: add durable `mission_runtime` and `preflight_run` records backed by DB, with Redis only as optional cache; expected impact: high robustness and better resumability; implementation complexity: medium; likely code areas affected: `backend/api/routes/routes_flights.py`, `backend/services/patrol/mission_runtime_store.py`, `backend/db/models.py`.
- [Medium] Smooth mission execution for survey legs; problem: `MavlinkDrone.follow_waypoints()` advances point-by-point with coarse sleeps and a simple distance threshold; why it matters: dense survey paths fly less smoothly and waste battery/time; proposed solution: replace it with a segment follower using lookahead, turn anticipation, configurable acceptance radii, and optional streamed setpoints for survey segments; expected impact: medium-high mission speed and path quality gain; implementation complexity: medium-high; likely code areas affected: `backend/drone/mavlink_drone.py`, `backend/flight/missions/grid_mission.py`.
- [Medium] Cache planner previews and make terrain sampling deterministic; problem: grid previews recompute too often and slope estimation still has random-sampling behavior; why it matters: planning UX will feel sluggish on large fields and operators will distrust small plan changes; proposed solution: hash mission inputs into a plan fingerprint, cache preview artifacts/stats, and replace random terrain sampling with deterministic grids or seeded samples; expected impact: high planning responsiveness and repeatability; implementation complexity: medium; likely code areas affected: `backend/flight/missions/grid_mission.py`, `backend/flight/missions/terrain_follow.py`, `backend/api/routes/routes_flights.py`, `frontend/src/pages/dashboard/tabs/PhotoGrammetry.tsx`.
- [Medium] Separate preview video from recording/processing; problem: the current MJPEG path forces server-side JPEG work and is not a premium live-video transport; why it matters: video becomes the first visible bottleneck during live missions; proposed solution: introduce a media abstraction layer now, keep MJPEG as fallback, and add WebRTC or FFmpeg/GStreamer-backed low-latency preview with a metadata side channel later; expected impact: high operator-perceived quality and lower CPU; implementation complexity: high; likely code areas affected: `backend/video/stream.py`, `backend/video/runtime.py`, `backend/api/routes/routes_video.py`, `frontend/src/components/dashboard/tasks/MissionVideoPanel.tsx`.
- [Quick win] Cut frontend polling and render churn; problem: live mission status still uses polling and some map-follow behavior is too eager, while large components recompute too much; why it matters: the UI will feel unstable exactly when the operator needs confidence; proposed solution: move live ops to websocket/React Query subscriptions, split the large planner/dashboard files, throttle auto-follow, and decimate telemetry for charts/maps; expected impact: high UI responsiveness; implementation complexity: medium; likely code areas affected: `frontend/src/pages/dashboard/tabs/TasksPage.tsx`, `frontend/src/pages/dashboard/tabs/PhotoGrammetry.tsx`, `frontend/src/hooks/useMissionStatusPolling.tsx`, `frontend/src/hooks/useDroneMapFollow.tsx`, `frontend/src/utils/CesiumMap.tsx`.
- [Quick win] Build replay-oriented read models; problem: analytics are currently summary-oriented, not operationally replayable; why it matters: premium drone products win on debrief, incident analysis, and proof-of-work; proposed solution: add flight timeline endpoints, event/telemetry downsampling, and mission summary tables generated in the background; expected impact: medium-high; implementation complexity: medium; likely code areas affected: `backend/api/routes/routes_analytics.py`, `backend/services/patrol/patrol_persistence.py`, `backend/services/photogrammetry/service.py`.

## 3. Product competitiveness
- As of March 31, 2026, the market baseline is clear: serious platforms offer live map/video, durable fleet/site operations, replay/debrief, compliance surfaces, and increasing levels of remote automation. That is visible in DJI FlightHub 2, DJI Dock 3, DroneDeploy Dock Automation, and Skydio Remote Flight Deck.
- Missing core capabilities for serious competition: persisted multi-flight runtime, mission templates/queues, low-latency video, replay/debrief, compliance-aware preflight, and a tighter fleet/site operations board. The repo has pieces of all of these, but not yet one cohesive operational product.
- The best commercial differentiators here are not "be another generic fleet dashboard." They are terrain-aware agriculture planning, a closed photogrammetry-to-action loop, and incident-aware operational workflows built from existing alerts/ML. Agriculture competitors increasingly monetize decision outputs such as prescription maps and targeted operations, not just imagery.
- Build now: unified runtime backbone, replay workspace, compliance-preflight package, planner caching/resume, and agriculture outputs. Build later: dock-grade remote autonomy, deep multi-tenant enterprise policy, and BVLOS-heavy workflows.
- Where this product can realistically win: agriculture, rural land management, utility vegetation/inspection surveys, and repeatable site operations where terrain-aware planning plus mapping outputs matter more than proprietary dock hardware. US compliance should be treated as table stakes, starting with FAA Remote ID and airspace/TFR awareness.

## 4. Features to add
### Mission features
- `Resumable survey missions`; user value: recover after battery swap, pause, or abort without reflighting completed swaths; business value: immediate time and battery savings; technical fit: strong, because `GridMission`, flight persistence, and runtime tracking already exist; MVP version: persist completed segment index and last-safe resume point per mission.
- `Mission templates and recurring site runs`; user value: one-click repeat missions for fields/sites; business value: higher operator throughput and stronger retention; technical fit: strong via existing mission schemas and planner preview; MVP version: save field, camera, altitude, overlap, speed, and pattern presets and queue them from the dashboard.

### Safety / compliance features
- `Compliance-aware preflight`; user value: fewer bad launches and better operator confidence; business value: credibility for commercial deployments; technical fit: extend existing preflight orchestration; MVP version: Remote ID status, geofence/TFR/NOTAM checks, and a go/no-go summary in preflight.
- `Alert playbooks and one-click emergency actions`; user value: faster response under weak link, battery, wind, or geofence events; business value: lower incident risk; technical fit: very strong because the alert engine already classifies these states; MVP version: map each alert severity to recommended actions and wire one-click pause/RTH/land.

### Operator workflow features
- `Live incident workspace`; user value: one place for active alerts, mission state, communications, and evidence; business value: better operational discipline and auditability; technical fit: use existing alerts, telemetry, video, and event persistence; MVP version: a single live-flight sidebar with alert timeline, notes, and command shortcuts.
- `Shift handoff / mission handover`; user value: smooth operator transitions on long jobs; business value: better support for real operations teams; technical fit: depends on durable mission runtime; MVP version: assign operator, leave mission notes, and reopen the exact live state.

### Agriculture-specific features
- `Prescription map export and zone-based missions`; user value: convert mapping outputs into actionable spray/spread workflows; business value: this is a real commercial wedge; technical fit: very strong on top of the current photogrammetry asset pipeline; MVP version: define manual treatment zones on orthomosaic/COG and export shapefile/GeoJSON prescriptions.
- `Temporal field comparison and coverage QA`; user value: compare field condition across dates and confirm survey/spray completeness; business value: repeat-use lock-in; technical fit: leverage stored orthomosaics and planned-vs-flown mission geometry; MVP version: field-to-field overlay with simple change heatmap and missed-area detection.

### Analytics / AI features
- `Flight replay with anomaly scoring`; user value: fast debrief and root-cause analysis; business value: premium differentiator and lower support cost; technical fit: high, using telemetry/events/video plus current alerts/ML; MVP version: scrubber-based replay with markers for alerts, link drops, wind spikes, and path deviations.
- `Image quality and coverage-gap QA`; user value: know immediately if a mapping run is usable; business value: fewer reflights and better trust; technical fit: natural extension of photogrammetry capture/session data; MVP version: blur/exposure heuristics, image-count checks, and overlap gap flags.

### Enterprise / fleet features
- `Fleet and site health board`; user value: one view for drone availability, battery state, last mission, incidents, and mapping jobs; business value: necessary for multi-site deployments; technical fit: good because telemetry, jobs, and persistence already exist; MVP version: per-site cards with current status, last flight, open alerts, and queued work.
- `Webhook/API automation`; user value: connect ERP/FMIS/work-order systems; business value: strong enterprise fit without overbuilding UI; technical fit: good, given the existing mission and mapping services; MVP version: outbound webhooks for mission state, alerts, and photogrammetry completion.

## 5. Visualization and UX improvements
- Most important first: a replay/debrief workspace that synchronizes map path, video, alerts, operator commands, and telemetry charts. That is the fastest path to a premium feel and a real operational differentiator; build from `backend/api/routes/routes_analytics.py`, `frontend/src/components/dashboard/tasks/MissionVideoPanel.tsx`, and `frontend/src/utils/CesiumMap.tsx`.
- Mission planning visuals should show camera footprint, overlap density, terrain/elevation profile, turn-radius stress points, wind-relative track direction, and completed-vs-unplanned swaths. The current planner already has the data model for this in `backend/flight/missions/grid_mission.py` and `frontend/src/pages/dashboard/tabs/PhotoGrammetry.tsx`.
- The live-flight dashboard should become an operator console, not a long page: large health cards for battery/link/GPS/video, a compact command strip, a map/video split view, and a visible mission-phase ladder. `frontend/src/pages/dashboard/tabs/TasksPage.tsx` and `frontend/src/components/dashboard/tasks/TelemetryHud.tsx` are the right starting points.
- Telemetry charts should be small, always-on ribbons with incident markers, not dense raw plots. Prioritize battery trend, link quality, altitude vs target, groundspeed, wind, and video latency; server-side decimation is mandatory before charting.
- Camera/video overlays should show gimbal heading, altitude AGL, speed, GPS quality, recording state, and mission leg ID. The current video panel is functional but visually thin; adding overlays will make even MJPEG feel more operational until transport is upgraded.
- Agricultural analytics layers should include orthomosaic/COG comparison, treatment zones, missed-coverage highlighting, and progress masks for live surveys. The repo already converts assets to usable geospatial formats, so the missing piece is UI composition, not core processing.
- Use 3D only where it helps decisions: Cesium terrain for slope/line-of-sight, photogrammetry 3D tiles for debrief/inspection, and a pseudo-3D elevation strip in planning. Avoid decorative 3D.

- Extend `backend/db/models.py` and `backend/db/repository/telemetry_repo.py` with batch append, downsample/materialized summary support, and tighter indexing on flight/time; why: replay and analytics need storage designed for reads; expected result: scalable telemetry/event persistence.

- Add plan fingerprinting, cached preview artifacts, and resume metadata in `backend/flight/missions/grid_mission.py` and `backend/flight/missions/terrain_follow.py`; why: planning must feel instant and repeatable; expected result: faster previews and more trustworthy plans.
- Rewrite the inner loop in `backend/drone/mavlink_drone.py` from waypoint blocking to segment following with better turn behavior; why: the current mission follower leaves efficiency on the table; expected result: smoother survey execution and better battery use.
- Split `frontend/src/pages/dashboard/tabs/PhotoGrammetry.tsx`, `frontend/src/pages/dashboard/tabs/TasksPage.tsx`, and `frontend/src/utils/CesiumMap.tsx` into feature modules backed by React Query/live stores; why: current UI size will slow every premium workflow; expected result: faster renders and easier evolution.

## 8. Risks and tradeoffs
- Unifying the telemetry/runtime backbone is the highest-value refactor, but it touches the most critical path. Mitigation: introduce it in shadow mode first and compare outputs before cutting over.
- Persisted runtime state adds schema and migration complexity, but not doing it keeps the product in "demo reliability" territory. This is worth the operational cost.
- Replay, downsampling, and richer analytics will increase storage volume. You need explicit retention and summary strategies, or the database will become the next bottleneck.
- Upgrading video to WebRTC or a richer media stack adds deployment complexity. It should follow telemetry/runtime cleanup, not precede it.
- Compliance and agronomy integrations create external dependencies and product-scope pressure. Keep the MVP narrow: preflight decision support first, decision-quality ag outputs second, full automation later.
- The right market tradeoff is vertical focus. If you optimize for agriculture/rural commercial ops, you gain a real wedge; if you chase every generic fleet use case, you will dilute the roadmap and still trail hardware-led platforms.
