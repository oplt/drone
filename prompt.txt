================================================================================
WAREHOUSE 3D MAP -> AISLE / RACK / SHELF / BIN EXTRACTION
How to auto-determine structure + inspection coordinates from a ready 3D map
================================================================================

GOAL
  After a warehouse_scan flight finishes and the 3D map is "ready", automatically
  determine the coordinates (+ approach poses) of aisles, racks, shelves and bins
  so the drone can be sent to each location to check products / scan barcodes,
  instead of an operator hand-placing every target by clicking the 3D viewer.


================================================================================
1. CURRENT FLOW (what exists today)
================================================================================

Scan -> Map:
  - plan_warehouse_scan() (planning/local_planner.py) generates a lawnmower /
    serpentine path from polygon_local_m (metres, dock-origin local frame).
    NOTE: the "aisle corridors" here are PRE-FLIGHT geometry from the polygon,
    NOT derived from the actual map.
  - fly_scan() (planning/scan.py) runs the flight; live-map bridges stream chunks.
  - On completion: build_manifest_from_flight_dir + validate_save_quality, then
    mapping_service.persist_capture() (service/mapping.py) marks the model "ready".

Artifacts available once "ready" (all in METRES, frame = odom == warehouse_map
when starting from the same dock):
  - Point-cloud chunks on disk under storage/warehouse-live-map/{flight_id}/:
      *.xyz32     (mono XYZ float32, e.g. mid360 raw lidar)
      *.xyzrgb32  (XYZ + RGB, e.g. rgbd_colored, nvblox_color/tsdf/esdf)
      *.glb       (nvblox mesh)
      *.meta.json (sidecar: source, layer, bbox_local_m[6], point_count, frame_id)
      live_map_manifest.json (chunk_counts, point_counts, map_quality, ...)
  - nvblox layers (the key ones for structure):
      /nvblox_node/static_esdf_pointcloud  -> nvblox_esdf  (distance-to-obstacle)
      /nvblox_node/tsdf_layer              -> nvblox_tsdf  (surfaces)
      /nvblox_node/color_layer             -> nvblox_color
      /nvblox_node/static_occupancy_grid / static_map_slice (NOT currently bridged)
  - DB: WarehouseModel/WarehouseMappingJob (status=ready), WarehouseAsset rows
    (ROSBAG / POINT_CLOUD / MESH_GLB / OCTOMAP / QUALITY_REPORT) with file paths.

Inspection (what exists):
  - WarehouseScanTarget (models.py:249) already models aisle_code, rack_code,
    shelf_level, bin_code, sku, barcode, product_name, target_point_local_json,
    scan_pose_local_json, shelf_normal_local_json, standoff_m, hover_time_s.
  - compute_scan_pose() (service/inspection.py:94) already turns a target point +
    shelf normal + standoff into a safe hover/approach pose with yaw.
  - build_inspection_waypoints() / execute_inspection_mission() already navigate
    target->target, hover, trigger scan, record results (MockWarehouseScanner).
  - Frontend: operator MANUALLY clicks the 3D viewer to drop bin targets
    (WarehouseLiveVoxelMapViewer.tsx) and fills aisle/rack/bin codes.

THE GAP (what does NOT exist anywhere):
  - No plane fitting / RANSAC, no clustering (DBSCAN/region grow), no occupancy
    grid analysis, no segmentation of the saved cloud.
  - No automatic aisle/rack/shelf/bin detection from the 3D map.
  - No automatic generation of WarehouseScanTarget rows or shelf normals.
  - No post-map re-planning: scan waypoints are computed once, pre-flight, from
    the polygon only.
  => Today, going from "3D map ready" to "coordinates to inspect" is 100% manual.


================================================================================
2. BEST APPROACH — automatic structure extraction pipeline
================================================================================

Run a post-flight "map -> structure" job that converts the saved cloud into a
hierarchy: warehouse -> aisles -> racks -> shelves -> bins, each with a centre
point, bounding box, and a shelf face normal (for approach pose). This is a
classic, well-bounded geometry problem on an axis-dominant environment.

Recommended layered algorithm (cheap -> rich; ship Stage A first):

STAGE A — Floor + vertical-structure separation (fast, deterministic)
  Input: merged downsampled cloud (voxel grid ~5 cm), prefer nvblox_tsdf/color
         (clean surfaces) or rgbd_colored; fall back to mid360 raw.
  1. Voxel-downsample the merged cloud (Open3D voxel_down_sample) for speed.
  2. Floor plane: RANSAC plane with normal ~ +Z (segment_plane). Gives floor z0
     and the working height band [z0+min, z0+max].
  3. Remove floor + ceiling; keep the vertical "rack mass".
  4. Project remaining points to the XY plane -> 2D occupancy/height map
     (e.g. 5-10 cm grid). Racks become thick lines; aisles become the empty gaps.

STAGE B — Aisle axis + aisle lanes
  1. Dominant axis: PCA on the 2D footprint OR reuse _dominant_axis_deg()
     (local_planner.py) / mission.aisle_axis_deg if the operator set it.
  2. Rotate footprint so racks are axis-aligned.
  3. Collapse occupancy onto the cross-axis -> 1D density profile. Peaks = rack
     rows; valleys (below threshold, width >= drone clearance from ESDF) = aisles.
  4. Each aisle lane = a free corridor centerline (start, end, width, axis).

STAGE C — Racks + bays/columns
  1. Within each rack row band, cluster points along the aisle axis (1D gaps /
     DBSCAN on the along-axis coordinate) -> rack segments / bays.
  2. Each rack: oriented bounding box (face plane = side facing the aisle).
     Face normal = horizontal normal pointing into the adjacent aisle (this is
     exactly the shelf_normal compute_scan_pose() needs).

STAGE D — Shelves (height levels) + bins
  1. Within a rack, histogram point density along Z -> horizontal shelf beams =
     density peaks => shelf_level z-values (regular spacing is a strong prior).
  2. Bins: divide each shelf face into regular cells along the aisle axis (use a
     configured bin pitch, or detect uprights as vertical density peaks).
  3. Each bin -> target_point_local (centre of the bin face) + the rack face
     normal + a default standoff -> WarehouseScanTarget.

OUTPUT of the job (write straight into existing schema):
  - One WarehouseScanTarget per bin (or per shelf if bins unknown):
      aisle_code   = "A{aisle_index}"        (or operator-mapped)
      rack_code    = "R{rack_index}"
      shelf_level  = level_index
      bin_code     = "B{bin_index}"
      target_point_local_json = bin face centre (x_m,y_m,z_m, frame warehouse_map)
      shelf_normal_local_json = rack face normal (into aisle)
      scan_pose_local_json    = compute_scan_pose(target, normal, standoff)
      standoff_m, hover_time_s, scan_timeout_s, priority = defaults/config
  - A structure summary JSON asset (aisles/racks/shelves with bboxes + centroids)
    saved as a WarehouseAsset (type e.g. "STRUCTURE_MAP") so the frontend can
    draw aisle/rack overlays and let the operator confirm/relabel.

WHY this fits the codebase:
  - Coordinates are already metric and in the same frame inspection uses
    (warehouse_map == odom from the same dock), so detected points are directly
    flyable via the existing inspection waypoint builder.
  - WarehouseScanTarget + compute_scan_pose + build_inspection_waypoints already
    consume exactly (target_point, shelf_normal, standoff) -> the detector only
    needs to PRODUCE those rows. Minimal new surface area.


================================================================================
3. PREFER ESDF/OCCUPANCY FOR NAVIGATION SAFETY
================================================================================

  - Use nvblox ESDF (distance field) to validate every generated scan pose has
    clearance >= drone radius + margin BEFORE saving it as a target (the current
    inspection MVP explicitly does NOT check clearance — api.py warns about this).
  - Bridge + persist the nvblox occupancy grid / static_map_slice (currently
    produced by nvblox but NOT bridged) to get a clean 2D free/occupied grid for
    aisle extraction and for A* routing between targets.
  - Reuse the existing OccupancyGrid.astar_path (planning/indoor/models.py) to
    order/route inspection targets along aisles (replace the greedy
    nearest_neighbor_targets() with aisle-aware, collision-free ordering).

4. WHERE THIS SHOULD RUN (architecture)
================================================================================

  - This is heavy, job-shaped, post-flight CPU work (downsample, RANSAC, cluster)
    => run it in the warehouse-mapping Celery worker, NOT inline in fly_scan and
    NOT on the API event loop.
  - warehouse_mapping_tasks.py is currently a STUB ("tasks were removed"); the
    celery queue/route warehouse_mapping.process_job already exists in
    entrypoints/workers/celery_app.py. Re-implement it:
        persist_capture (map ready) -> enqueue warehouse_mapping.extract_structure
        -> writes WarehouseScanTarget rows + STRUCTURE_MAP asset -> notifies UI.
  - Keep it idempotent + re-runnable (operator can re-extract with different bin
    pitch / clearance without re-flying).
  - Libraries: Open3D (voxel/RANSAC/DBSCAN/OBB) or numpy+scipy (KDTree, label).
    Use GPU only if a batch justifies it; CPU Open3D is fine for one warehouse.

5. SUGGESTED IMPLEMENTATION ORDER (incremental, each independently useful)
================================================================================

  P0  Loader: merge {flight_id} chunks (.xyz32/.xyzrgb32 via pointcloud2_parser
      decoders) + voxel-downsample into one numpy array in warehouse_map frame.
  P0  Stage A+B: floor removal + aisle extraction -> aisle centerlines + a
      STRUCTURE_MAP asset; render aisle overlays in the 3D viewer for confirmation.
  P1  Stage C+D: rack/shelf/bin segmentation -> auto-create WarehouseScanTarget
      rows (operator confirms/edits codes + SKUs in the existing panel).
  P1  ESDF clearance gate on generated scan poses (safety before any auto-fly).
  P2  Bridge + persist nvblox occupancy grid; aisle-aware A* target routing.
  P2  Re-run/refine endpoint (bin pitch, standoff, clearance as params).
  P3  Optional learned refinement (only if geometric heuristics prove too noisy):
      train a small classifier on labelled shelves; keep geometry as the prior.

7. IMPLEMENTED (this change) — what now exists
================================================================================

Backend (runs in the warehouse-mapping Celery worker, never on the API loop):
  - NEW service/structure_extraction.py  — pure numpy/scipy pipeline:
      P0 load_flight_cloud(): merge {flight_id} .xyz32/.xyzrgb32 chunks (ESDF/mesh
         excluded) + voxel-downsample into one (N,3) array in warehouse_map frame.
      A  floor/ceiling removal via z-histogram -> vertical rack mass.
      B  PCA dominant aisle axis -> cross-axis density profile (scipy.ndimage.label)
         -> rack rows (dense) + aisle lanes (free gaps >= min_aisle_width).
      C  along-axis density split of each rack row -> rack bays + face normal
         (horizontal, pointing into the nearest adjacent aisle).
      D  z-histogram -> shelf levels; bin pitch -> bins -> target_point + scan_pose
         (reuses inspection.compute_scan_pose, no change there).
      + clearance gate: cKDTree distance from every scan pose to the rack mass must
        be >= drone_radius + clearance_margin, else the target is dropped (counted).
      + aisle-aware serpentine ordering -> WarehouseScanTarget.priority (boustrophedon
        per aisle) so inspection flies the shortest sweep without full A*.
  - NEW service/structure_jobs.py        — async orchestration: runs the geometry in
      a thread, idempotently replaces auto targets for a model
      (reference_model_id == model_id; operator/manual targets preserved), writes a
      STRUCTURE_MAP WarehouseAsset (summary JSON: floor_z, axis_deg, aisles[], racks[]
      with bboxes + face codes, counts, params) next to the flight chunks.
  - entrypoints/workers/warehouse_mapping_tasks.py — task warehouse_mapping.extract_structure
      (queue warehouse-mapping), routed in celery_app.py. Re-runnable with param overrides.
  - service/mapping.py:persist_capture — enqueues extraction automatically once a map
      becomes "ready" (gated by warehouse_structure_extraction_enabled; best-effort).
  - api.py:
      POST /warehouse/maps/{id}/structure/extract  (re-run; optional bin_pitch/standoff/
           clearance/voxel/axis overrides) -> 202 queued.
      GET  /warehouse/maps/{id}/structure          -> latest STRUCTURE_MAP summary.
  - runtime.py: warehouse_structure_* tunables (voxel, grid res, floor margin, aisle
      width, bin pitch, standoff, drone radius, clearance margin, max points).

Frontend:
  - api/warehouseInspectionApi.ts: fetchWarehouseStructure / extractWarehouseStructure + types.
  - hooks/useWarehouseStructure.ts: loads the latest structure, polls passively, and
      exposes extract() that resolves once a *newer* structure is persisted.
  - WarehouseLiveVoxelScene.tsx: StructureOverlay draws aisle centrelines (cyan dashed)
      + rack wireframe boxes (purple, rotated by axis_deg) over the point cloud.
  - WarehouseCoordinateSetupPanel.tsx: "Auto-detect warehouse plan" button (+ bin pitch),
      shows aisle/rack/target counts, then refreshes the editable target table so the
      operator confirms/edits codes, SKUs and barcodes on the auto-generated rows.

Still open (noted, not done here — needs ROS runtime, not a pure code change):
  - Bridging/persisting the nvblox static occupancy grid for full A* routing; the
      current ordering is aisle-serpentine via priority. ESDF clearance is approximated
      with the cloud KD-tree instead of the nvblox distance field.


================================================================================
8. HOW TO ASSIGN A "WAREHOUSE PLAN"
================================================================================

A "warehouse plan" = the detected structure (STRUCTURE_MAP asset) + the set of
WarehouseScanTarget rows, both assigned to a warehouse_map_id and tied to the
WarehouseModel version they were derived from (reference_model_id). It is what
turns a raw 3D map into a flyable inspection program.

Lifecycle / operator workflow
  1. Fly a warehouse_scan mission. On completion persist_capture() marks the model
     "ready" and AUTO-ENQUEUES warehouse_mapping.extract_structure for that flight.
     (If warehouse_structure_extraction_enabled is False, skip to step 3.)
  2. The worker builds the plan: it writes the STRUCTURE_MAP asset and the
     auto-generated WarehouseScanTarget rows (priority = serpentine order, all
     active=True, reference_model_id = the ready model).
  3. Operator opens the map -> "Coordinate Setup" tab. Aisles (cyan) and racks
     (purple) are overlaid on the cloud. To (re)generate or tune, click
     "Auto-detect warehouse plan" (optionally set bin pitch); this calls
     POST /warehouse/maps/{id}/structure/extract and reloads the target table.
  4. Operator CONFIRMS the plan: review/edit the generated rows (aisle/rack/bin
     codes, SKU, barcode, product, standoff, hover) inline, archive false positives,
     or hand-place extra bins with the existing pick-on-map flow. This human pass is
     the "assignment": the geometry proposes, the operator ratifies.
  5. Assign the plan to an inspection run: select targets and create a mission via
     POST /warehouse/inspection-missions (warehouse_map_id + target_ids + scan_mode,
     optimize_order=True). build_inspection_waypoints consumes the assigned targets'
     scan_pose/shelf_normal/standoff directly; results persist per target.

Assignment rules / guarantees
  - Scope: a plan belongs to exactly one warehouse_map_id; targets and the
    STRUCTURE_MAP asset are filtered by it (ownership enforced via _get_map_or_404).
  - Provenance: auto targets carry reference_model_id = the model they came from.
  - Idempotent re-assignment: re-running extraction for the same model replaces only
    that model's auto targets (reference_model_id == model_id). Manual targets
    (reference_model_id NULL) and targets from other models are never touched, so an
    operator can re-detect without losing hand edits made on a different basis.
  - Re-tuning: change bin pitch / standoff / clearance / axis via the extract
    endpoint params to re-assign a denser/sparser plan; no re-flight required.
  - Versioning: a new scan creates a new WarehouseModel version -> a new plan; the
    previous plan's targets remain until explicitly archived, so plans are
    comparable across scans (reuse /scanned-maps/compare for the underlying maps).
  - Safety: only poses passing the clearance gate are assigned, so an assigned plan
    is collision-light by construction; the serpentine priority defines fly order.
