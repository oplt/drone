# Coordinate Setup Analysis for Indoor Warehouse Drone Mission

## Executive Summary

The repository has a substantial indoor-mapping foundation, but coordinate handling is not yet safe enough for repeatable inspection missions.
ROS/Gazebo mapping data is currently normalized into `odom`; scan targets are labeled `warehouse_map`; and the inspection workflow explicitly assumes `warehouse_map == odom`.
This works only when every mission starts from exactly the same physical pose and heading as the mapping run. No persisted, versioned `warehouse_map -> odom` localization transform exists.

The strongest existing pieces are the ROS 2/Gazebo sensor bridge, `tf2` point-cloud transforms, nvBlox/occupancy ingestion, metric local mission planning, automatic rack/aisle extraction, scan-target persistence, and the Three.js coordinate-picking UI.
The largest gaps are:

1. no authoritative coordinate-frame record or locked map origin;
2. no persistent warehouse-layout hierarchy (aisle/rack/shelf/bin are strings or extraction JSON);
3. no coordinate/layout version pinned to missions;
4. no complete pre-execution ESDF/path validation for inspection missions;
5. frontend scene coordinates are treated as warehouse coordinates without checking chunk/frame alignment;
6. live mapping uses `odom`, while saved targets use `warehouse_map`;
7. current `backend/modules/warehouse/routers/scan_targets.py` contains naming regressions (`scanpose_*`, `dronepose_*`, `compute_scanpose`) that break target create/update/pose computation/mock execution at runtime.

Recommended architecture: persist all warehouse objects in a stable `warehouse_map` ENU, right-handed, metre/radian coordinate system;
maintain `map -> odom -> base_link -> sensor` through `tf2`; treat `odom` as live and drifting;
convert to MAVLink NED only inside the vehicle adapter; and require a validated, locked coordinate version before mission execution.

Analysis basis: the existing `$understand` graph covers 878 files at commit `2fefe6dc3bddb9dfb44996645cd52051344b3eca` (878 nodes, 8,008 edges, 14 layers).
The working tree contains uncommitted changes, so current source files were inspected directly and take precedence over graph summaries. No source code was modified.

## Current Coordinate Flow

### Project architecture and relevant modules

- FastAPI/SQLAlchemy backend: `backend/modules/warehouse`, `backend/modules/missions`, `backend/modules/vehicle_runtime`, `backend/infrastructure/vehicle`.
- React/TypeScript/Three.js frontend: `frontend/src/modules/warehouse`.
- ROS 2 Jazzy/Gazebo bridge: `ros2_ws/src/drone_gz_bridge`.
- Warehouse persistence: `backend/modules/warehouse/models.py`, repositories under `backend/modules/warehouse/repository`, Alembic migrations under `backend/infrastructure/persistence/alembic/versions`.
- Mapping artifacts: `WarehouseModel`, `WarehouseMappingJob`, and `WarehouseAsset`; live chunks are stored by `service/live_map_storage.py` and described by live-map manifests.

### Warehouse creation and boundary

`POST /warehouse/maps` in `backend/modules/warehouse/routers/maps.py` accepts only name, width, and length (`WarehouseMapCreateIn` in `http_models.py`).
It creates an axis-aligned rectangle from `(0,0)` to `(width,length)`. `repository/maps.py` validates finite polygon points and Shapely polygon area,
then stores `polygon_local_m` inside `WarehouseMap.meta_data`.
`WarehouseMap.boundary` and `centroid` are nullable WGS84 GeoAlchemy fields and remain `None` for indoor maps.

The UI text in `frontend/src/modules/warehouse/views/Warehouse.tsx` states that origin `(0,0)` is the takeoff position.
This is a convention, not a persisted coordinate-frame definition.
Warehouse responses expose `polygon_local_m`, but not frame ID, units, axis convention, origin pose, transform, lock status, or version.

### ROS/Gazebo frame flow

`ros2_ws/src/drone_gz_bridge/config/warehouse_bridge.yaml` bridges:

- `/warehouse/drone/odometry` (`nav_msgs/Odometry`),
- `/scan`, `/scan/points`, `/warehouse/mid360/points`,
- RGB-D images, depth, points, camera info,
- stereo images, `/warehouse/imu`, joint states, and gimbal commands.

`drone_gz_bridge/odom_to_tf.py` republishes each odometry message directly as a TF using its supplied `header.frame_id` and `child_frame_id`.
This is correct as a transport adapter but does not enforce expected names, reject self/cyclic transforms, or detect duplicate broadcasters.

`launch/warehouse_bridge.launch.py` defines:

- body: `iris_with_standoffs/base_link`;
- LiDAR: `iris_rplidar_rgbd/mid360_lidar_link/mid360_lidar`;
- RGB-D: `iris_rplidar_rgbd/front_rgbd_camera_link/front_rgbd_camera`;
- IMU: `imu_link`.

It publishes static transforms with hard-coded translations and zero rotations.
Sensor URDF/SDF calibration is not the source of truth.
Frame names are long Gazebo model paths rather than stable robot frame names.

`launch/warehouse_nvblox.launch.py`, `backend/core/config/runtime.py`, `service/mapping_stack_lifecycle.py`, and `service/map_source_config.py` use `global_frame:=odom` and `pose_frame:=iris_with_standoffs/base_link`. Thus all mapped point clouds, occupancy, ESDF, TSDF, color voxels, and mesh products are currently aligned to `odom`, not a durable map frame.

`raw_pointcloud_live_map_bridge.py`, `colored_pointcloud_live_map_bridge.py`, and `nvblox_layers_live_map_bridge.py` use `tf2` lookups to transform sensor data into each source's configured global frame. This is good integration-boundary behavior. Failed lookups are logged and frames are attached to chunks. However, defaults elsewhere conflict: `WarehouseLiveMapUpdate` and `WarehouseLivePose` in `live_map_stream.py` default to `map`, while the ROS bridges publish `odom`.

`live_map_diagnostics.py`, `live_map_readiness.py`, and `sim_time_tf_readiness.py` check the `odom -> iris_with_standoffs/base_link` chain and sensor availability. They do not require `warehouse_map -> odom` because that transform does not exist.

### Live scan and scan artifact flow

Live ROS data is parsed by:

- `pointcloud2_parser.py` for XYZ/RGB point clouds;
- `occupancy_grid_parser.py` for occupancy metadata, resolution, origin, and cells;
- `nvblox_mesh_adapter.py` for mesh conversion;
- raw/colored/nvBlox live-map bridges for frame transformation and chunk publication.

`live_map_stream.py` publishes per-flight `WarehouseLiveMapUpdate` objects containing a frame ID, drone pose, changed chunks, scan-path samples, and health including drift estimate. `live_map_storage.py`, `live_map_manifest.py`, `live_map_snapshot_cache.py`, and `live_map_replay.py` persist/replay chunk-level scan results. `routers/scanned_maps.py` lists finalized `WarehouseMappingJob`/`WarehouseModel` results and assets.

`service/mapping.py` converts scan artifacts into `WarehouseAsset` rows. `WarehouseAsset.meta_data` can carry artifact metadata, but there is no normalized transform/provenance contract binding every artifact to an immutable coordinate-frame version.

### Scan-to-layout extraction

`service/structure_extraction.py` loads saved point clouds, downsamples them, detects a floor and height band, estimates a dominant warehouse axis, segments rack mass, derives aisle centerlines/rack boxes, synthesizes shelf/bin scan targets, and checks candidate scan poses using KD-tree/occupancy clearance. It writes `structure_map.json` through `structure_jobs.py` and creates `WarehouseScanTarget` rows.

The extraction file documents that `warehouse_map` is assumed equal to `odom` from the same takeoff origin. Results include `frame_id`, `floor_z`, `axis_deg`, `centerline_world`, `center_world`, dimensions, counts, confidence, rejection reasons, and extraction parameters. Aisles/racks are not persisted as first-class relational entities; only summary JSON and generated scan-target label strings survive.

The process is post-scan. During live scanning, overlays and chunks update, but there is no persistent provisional object model, user-confirmation state, or confidence history for partial aisle/rack observations.

### Scan-target and inspection mission flow

`WarehouseScanTarget` in `models.py` stores aisle/rack/shelf/bin labels, target point JSON, scan pose JSON, shelf normal JSON, standoff, hover/timeout, priority, and active status. Pydantic types in `schemas.py` use metres and default `frame_id="warehouse_map"`. They validate finite values, yaw range, nonzero normals, and matching frame IDs.

`service/inspection.py` computes standoff poses, normalizes yaw, orders targets, and creates four logical mission steps per target. Ordering is a 2D greedy nearest-neighbor heuristic after label/priority sorting. It does not use the occupancy grid, ESDF, kinematic constraints, approach corridors, or return energy.

`POST /warehouse/inspection-missions` in `routers/scan_targets.py` persists waypoints in `WarehouseInspectionMission.plan_json`. It explicitly saves:

- `frame_id: warehouse_map`;
- `warehouse_map_to_odom_transform: null`;
- warnings that ESDF validation is not wired and `warehouse_map == odom` is assumed.

`backend/modules/warehouse/docs/warehouse-inspection.md` confirms this MVP constraint. Real inspection execution exists through `LocalNavigationAdapter` in `service/inspection.py`, but the exposed route shown here includes a mock scanner runner.

Separate warehouse scan missions use `planning/mission.py`, `planning/local_planner.py`, and `planning/scan.py`. They build local metric coverage paths relative to dock/takeoff origin, apply polygon inset and clearance rules, and execute a larger indoor state machine. Unknown-space exploration uses typed `LocalPose`, `OccupancyGrid`, `IndoorFrame` (`map`, `odom`, `base_link`, `dock`), frontier planning, and a `SLAMProvider`; `SimulatedSLAMProvider` still implements transforms by relabeling/offset assumptions rather than a real `tf2`/localization adapter.

Outdoor mission paths use latitude/longitude through `backend/modules/missions` and `backend/infrastructure/vehicle/mavlink_client.py`. This path is distinct from warehouse local coordinates. There is no central ENU/NED conversion service joining indoor warehouse waypoints to MAVLink local setpoints.

### Frontend coordinate interaction

`WarehouseLiveVoxelMapViewer.tsx` hosts the Three.js scene and Coordinate Setup tab. `WarehouseLiveVoxelScene.tsx` renders chunks, drone path, structure overlays, targets, poses, and a 240 m picking plane. `MapPickPlane` copies `event.point.x/y` directly into `x_m/y_m` and substitutes user-selected `placementZ`. No explicit frontend-frame transform or frame compatibility check is applied.

`useWarehouseMapPlacement.ts` loads saved targets and manages pick mode. `useWarehouseScanPoseDraft.ts` calls `/warehouse/scan-targets/compute-scan-pose`. `WarehouseCoordinateSetupPanel.tsx` lets the operator pick one bin/target point, select one of four axis-aligned shelf normals, enter labels/standoff/hover time, save, list, select, and archive targets. It can trigger automatic structure extraction and shows quality/confidence.

The frontend does not currently provide origin placement, arbitrary orientation, aisle/rack/shelf geometry editing, boundary editing, drag handles, undo/redo, snapping, hierarchy management, no-fly/safety shapes, coordinate validation, mission-path preview, or 2D/3D synchronized editing.

## Problems and Risks

### 1. `warehouse_map` is only an alias for drifting `odom`

- Severity: Critical
- Affected: `backend/modules/warehouse/docs/warehouse-inspection.md`, `service/structure_extraction.py`, `service/map_source_config.py`, `routers/scan_targets.py`, `ros2_ws/.../warehouse_nvblox.launch.py`.
- Current behavior: scan artifacts are produced in `odom`; saved targets claim `warehouse_map`; mission transform is `null`.
- Risk: restart, relocalization, loop closure, changed takeoff pose, or accumulated odometry drift moves every rack/bin waypoint relative to reality.
- Fix: persist a versioned `WarehouseCoordinateFrame`; publish/resolve `warehouse_map -> odom`; require localization confidence and a locked transform before execution.

### 2. Scan-target router has runtime-breaking naming regressions

- Severity: Critical
- Affected: `backend/modules/warehouse/routers/scan_targets.py` lines using `scanpose_local_json`, `dronepose_local_json`, `compute_scanpose`, and `scanpose`.
- Current behavior: model/schema/service names are `scan_pose_local_json`, `drone_pose_local_json`, `compute_scan_pose`, and response field `scan_pose`; router uses nonexistent alternatives in create, update, compute, and mock-run paths.
- Risk: target creation/update and pose computation raise attribute/name/constructor errors; mock result persistence also fails.
- Fix: restore canonical names and add API integration tests for create, patch, compute pose, mission create, and mock execution before coordinate feature work.

### 3. No authoritative coordinate-frame model

- Severity: Critical
- Affected: `WarehouseMap`, `WarehouseModel`, `WarehouseAsset`, `WarehouseInspectionMission`, API response models.
- Current behavior: frame/origin/units live in conventions and JSON fragments.
- Risk: silent frame and unit mismatches cannot be queried, constrained, migrated, audited, or pinned.
- Fix: normalized coordinate-frame/version tables with units, axes, quaternion transform, source, status, covariance, timestamps, and immutable version IDs.

### 4. Live-map default frame conflicts with bridge frame

- Severity: High
- Affected: `service/live_map_stream.py`, all live-map bridges, frontend live-map state/tests.
- Current behavior: stream defaults to `map`; configured bridge global frame is `odom`; tests exercise both.
- Risk: missing frame fields silently relabel `odom` coordinates as `map`.
- Fix: remove permissive defaults at ingestion, require explicit frame IDs, reject mixed-frame updates unless transformed.

### 5. UI scene coordinates leak into mission data

- Severity: High
- Affected: `WarehouseLiveVoxelScene.tsx`, `useWarehouseMapPlacement.ts`, `warehouseMapPlacement.ts`.
- Current behavior: Three.js `event.point` becomes backend metric coordinates directly.
- Risk: any viewer rotation, model root transform, artifact scale, axis swap, or frame mismatch corrupts targets.
- Fix: explicit `sceneToWarehouseMap()`/`warehouseMapToScene()` using a displayed frame/version; disallow picking when visible layers disagree.

### 6. Warehouse hierarchy is denormalized labels

- Severity: High
- Affected: `WarehouseScanTarget`, extraction summary JSON, coordinate panel.
- Current behavior: aisle/rack/bin are strings; shelf is an integer; geometry has no independent identity or parent constraints.
- Risk: typos split entities, geometry cannot be edited coherently, duplication is uncontrolled, and targets cannot inherit transforms/dimensions.
- Fix: versioned `Aisle`, `Rack`, `Shelf`, `Bin`, `InspectionTarget`, and `SafetyZone` entities with foreign keys.

### 7. No coordinate/layout version lock on missions

- Severity: High
- Affected: `WarehouseInspectionMission.plan_json`, `WarehouseModel.version`, scan targets.
- Current behavior: mission references target IDs but no immutable layout/frame revision.
- Risk: edits after planning can make preview differ from execution.
- Fix: mission stores `coordinate_frame_version_id`, `layout_version_id`, `map_model_id`, artifact checksums, and validation-result ID; execution rejects stale versions.

### 8. Inspection path validation is incomplete

- Severity: Critical
- Affected: `service/inspection.py`, `routers/scan_targets.py`.
- Current behavior: scan-pose candidates may be checked during extraction, but manual targets and paths between them are not checked against current ESDF/occupancy; mission warning admits this.
- Risk: collision with racks, ceilings, transient obstacles, or unsafe approach direction.
- Fix: unified validation service for point clearance, swept-volume path, approach cone, altitude, geofence, return path, and map freshness.

### 9. Sensor extrinsics have competing sources and unstable names

- Severity: High
- Affected: `warehouse_bridge.launch.py`, `WarehouseSensorRig`, Gazebo SDF/URDF, diagnostics.
- Current behavior: hard-coded static TF values coexist with stored extrinsics URLs/JSON; body/sensor names include simulation model paths.
- Risk: map deformation, duplicated TF broadcasters, and sim/real inconsistency.
- Fix: stable REP-105/REP-103 names (`base_link`, `lidar_link`, `camera_link`, optical frames, `imu_link`), one generated calibration source, checksum validation, and launch-time TF assertions.

### 10. Origin and boundary cannot be edited or locked

- Severity: High
- Affected: `WarehouseMapCreateIn`, `routers/maps.py`, `WarehouseMapOut`, setup UI.
- Current behavior: axis-aligned rectangle and takeoff-origin convention only.
- Risk: scanned warehouse orientation/shape cannot be reconciled with operator intent; later origin changes invalidate all children silently.
- Fix: coordinate-frame setup endpoints, arbitrary polygon editor, origin pose, north/aisle alignment, lock workflow, and migration/transform preview.

### 11. Extraction output is not traceable enough

- Severity: Medium
- Affected: `structure_extraction.py`, `structure_jobs.py`, `WarehouseAsset.meta_data`.
- Current behavior: summary captures parameters and confidence, but layout objects are generated/replaced without an explicit immutable input-frame/checksum/version lineage.
- Risk: difficult reproduction and unsafe re-extraction over manual corrections.
- Fix: `ScanArtifactSet` and `LayoutVersion` provenance; mark objects auto/manual/confirmed; never overwrite confirmed edits.

### 12. Orientation representation is too weak

- Severity: Medium
- Affected: `WarehouseLocalPose`, dock HTTP model, scan targets.
- Current behavior: position plus yaw only; dock models in `http_models.py` omit frame ID; gimbal pitch/roll is not part of inspection pose.
- Risk: ambiguous camera optical orientation and insufficient 3D approach control.
- Fix: canonical quaternion pose plus derived yaw/pitch/roll; separate vehicle pose and sensor/gimbal aim constraints.

### 13. Validation permits arbitrary but matching frame strings

- Severity: High
- Affected: `schemas.py`.
- Current behavior: target, scan pose, and normal must share a nonblank frame, but any string passes.
- Risk: `foo`, `odom`, or sensor-frame targets can enter a warehouse-map mission without transformation.
- Fix: require a registered frame/version; for layout APIs accept only the warehouse frame and transform imports explicitly.

### 14. Audit and diagnostics are incomplete

- Severity: Medium
- Affected: warehouse routers/services and observability modules.
- Current behavior: mission and mock scan events are logged, but origin/layout edits and transform decisions are not first-class audit events.
- Risk: coordinate incidents cannot be reconstructed.
- Fix: structured audit events with old/new values, actor, reason, frame/version, transform age, covariance, and validation result.

## Coordinate Frame Improvements

Adopt this canonical TF tree and data contract:

`warehouse_map -> odom -> base_link -> {lidar_link, camera_link, camera_optical_frame, rgbd_link, imu_link, gimbal_link}`

- `warehouse_map`: stable ENU-style local Cartesian frame. +X along confirmed warehouse primary axis, +Y left in the horizontal plane, +Z up. Units metres; right-handed. All warehouse layout geometry is stored here.
- `odom`: continuous local motion frame. Smooth but allowed to drift. Never used as persistent object storage.
- `base_link`: drone body frame following ROS REP-103. If flight-control body convention differs, expose a dedicated `base_link_frd` transform rather than overloading `base_link`.
- Sensor frames: calibrated static/dynamic TF children. Camera optical frame follows optical convention; gimbal joints are dynamic transforms.
- `dock`: optional semantic frame attached to `warehouse_map`; do not insert it between map and odom.
- Frontend frame: Three.js scene may use the same XYZ convention, but conversion functions and a frame/version ID remain mandatory.
- Mission frame: plan/store in `warehouse_map`; resolve to live control frame immediately before dispatch.

Persist `WarehouseCoordinateFrameVersion` fields: ID, warehouse map ID, semantic name, parent/reference frame, units, axis convention, origin translation, origin quaternion, source scan/model, covariance, localization method, status (`draft`, `validated`, `locked`, `superseded`), version, checksum, creator, timestamps.

ENU/NED boundary rules:

- Backend domain/layout uses ENU metres and radians internally.
- ROS uses ENU/right-handed frames through TF.
- MAVLink conversion occurs only in `backend/infrastructure/vehicle/mavlink_client.py` or a dedicated adapter: ENU `(x,y,z)` to local NED `(north=y, east=x, down=-z)` after applying the current map-to-autopilot transform. Yaw conversion must be centralized and tested; never swap axes in UI or mission services.
- Gazebo pose must be validated against ROS bridge output with known landmark tests. Do not assume model/world axes merely because current simulation appears aligned.

Drift/mismatch prevention:

- require transform timestamp, maximum age, covariance, and localization confidence;
- log map/odom deltas and alarm on jumps;
- freeze coordinate version for active missions;
- require explicit metres/radians at internal boundaries and degrees only in API display fields where named `_deg`;
- reject mixed-frame chunks and coordinates;
- record map resolution/scale and verify known-distance calibration targets;
- use `tf2_ros.Buffer.lookup_transform()` at acquisition/control boundaries, never manual relabeling.

## 3D Scan Coordinate Setup

### After Scan Completion

1. Finalize an immutable artifact set containing point cloud, mesh, occupancy grid, ESDF/TSDF, trajectory, frame IDs, timestamps, scale, resolution, checksums, and calibration hash. Extend `WarehouseAsset.meta_data` or preferably add `WarehouseScanArtifactSet`.
2. Transform all artifacts into the selected stable `warehouse_map` frame. If scan was produced in `odom`, estimate and persist `warehouse_map -> scan_odom` rather than renaming it.
3. Detect floor plane with RANSAC/robust fitting, align +Z, report residual and inlier ratio. Existing floor logic in `structure_extraction.py` can seed this stage.
4. Estimate dominant wall/aisle axes. Present alternative 90-degree orientations and allow operator flip/rotation before locking.
5. Select origin from dock marker, surveyed fiducial, or operator-picked floor point. Takeoff position may be a default, not an implicit permanent origin.
6. Validate scale using known rack spacing, marker size, stereo baseline, or surveyed distance. Reject non-metre or anisotropic scale.
7. Run current rack/aisle extraction, but persist candidates as draft layout entities with confidence and source geometry.
8. Let users adjust floor, origin, axis, boundary, racks, shelves, and bins. Recompute dependent coordinates through a proposed transform preview.
9. Save a locked coordinate-frame version and a layout version. Link every object and scan artifact to both.
10. Run topology, containment, overlap, clearance, reachability, and path validation before marking layout `mission_ready`.

### During Live Scan

- Keep live coordinates in a provisional scan frame tied to current SLAM/odom epoch.
- Publish revisioned live-map updates: frame ID, transform epoch, sequence, confidence, and pose covariance.
- Detect provisional floor/aisle/rack candidates incrementally. Store observations separately from confirmed objects.
- Show confidence heatmaps, stale regions, drift, loop-closure updates, and candidate movement in `WarehouseLiveVoxelMapViewer`.
- Permit user labels, but attach them to map features/landmarks, not raw transient coordinates. Reproject labels after SLAM optimization.
- Use states `candidate -> user_labeled -> confirmed -> locked`; retain source observations and confidence history.
- Do not permit executable missions from provisional coordinates. Preview may be allowed with a prominent non-executable status.
- At scan completion, run global optimization, freeze artifact transforms, show displacement from provisional coordinates, require operator confirmation when thresholds are exceeded, then create locked versions.

## Manual Coordinate Setup for Warehouse Objects

Adapt the suggested schema into version-owned relational tables. Every object includes `id`, `layout_version_id`, `frame_version_id`, `source` (`manual`, `scan_detected`, `imported`), confidence, review state, metadata, created/updated actor/time, and optimistic-lock revision.

### Warehouse / Layout Version

- Fields: name, boundary polygon with floor/ceiling, coordinate-frame version, units (`m` only), origin pose, scale (`1.0`), status, scan-model/artifact reference.
- Validation: valid simple polygon, positive area/height, locked frame for mission-ready status.
- UI: 2D/3D boundary and origin handles; axis gizmo; lock/unlock with impact preview.
- API: GET/PUT coordinate frame; create draft layout version; validate; publish/lock.

### Aisle

- Fields: code/name, centerline or polygon, width, floor/height limits, directionality, preferred travel direction, speed limit, safety margin.
- Validation: inside warehouse; positive width; navigable after safety inset; no forbidden intersections.
- UI: draw line/rectangle, drag endpoints, width handles, align/snap/duplicate.
- API: batch CRUD under `/warehouse/maps/{id}/layout-versions/{version}/aisles`.

### Rack

- Fields: aisle ID, code/name, oriented 3D box, pose quaternion, dimensions, side/faces, safety distance.
- Validation: belongs to/intersects allowed aisle side, inside boundary, non-overlap unless explicitly joined.
- UI: place oriented box, rotate, resize, assign aisle/side, array duplicate.
- API: batch CRUD with server-side containment/overlap response.

### Shelf

- Fields: rack ID, level, local pose relative to rack, dimensions, height from floor, load/visibility metadata.
- Validation: contained in rack, unique level/slot, positive dimensions, monotonic height where configured.
- UI: vertical rack elevation editor, evenly distribute levels, duplicate levels.
- API: nested or layout-level CRUD; backend resolves rack-local to warehouse-map coordinates.

### Bin

- Fields: shelf ID, label, shelf-local oriented box, barcode/QR/SKU, preferred face, inspection target reference.
- Validation: contained in shelf, unique label per shelf, non-overlap, readable face.
- UI: grid subdivision, multi-create, barcode import, per-bin marker.
- API: batch upsert/import with per-row errors and dry-run.

### Inspection Point / Pose

- Fields: target object ID, vehicle pose, camera/gimbal aim, standoff, approach corridor, hover time, scan mode, tolerance.
- Validation: registered frame, clear swept volume, line of sight, reachable altitude/gimbal range, inside allowed airspace.
- UI: target/pose pair, approach arrow/cone, camera frustum, clearance badge.
- API: compute candidates, validate, accept override with reason.

### No-fly Zone and Safety Boundary

- Fields: zone type, 2D polygon extrusion or 3D volume, floor/ceiling, active schedule, margin, severity.
- Validation: valid volume, coordinate version, no contradiction with mandatory takeoff/dock corridor.
- UI: draw polygon/box, set vertical range, distinct warning color.
- API: CRUD plus mission intersection query.

### Charging Station and Takeoff/Landing Point

- Extend existing `WarehouseDockStation`: explicit frame/version, full pose quaternion, approach/exit corridors, pad dimensions, marker transform/covariance, precision requirement.
- Add semantic point type for emergency landing zones where no charger exists.
- Validate clearance, marker visibility, approach direction, and return reachability.

## Backend Improvements

Add services under `backend/modules/warehouse/service`:

- `coordinate_transform.py`: typed pose/vector/geometry transforms using registered frame versions; ROS adapter uses `tf2`, pure math layer uses homogeneous transforms.
- `coordinate_validation.py`: finite/unit/frame/boundary/hierarchy/overlap/clearance/version rules with structured error codes.
- `layout.py`: version lifecycle and aisle/rack/shelf/bin CRUD.
- `scan_to_layout.py`: wraps `structure_extraction.py`, creates draft entities, preserves manual confirmations.
- `inspection_planner.py`: converts layout targets to safe approach/hover/exit waypoint sequences and calls occupancy/ESDF planner.
- `coordinate_import_export.py`: versioned JSON/GeoJSON-like local schema with dry-run and checksums.
- `coordinate_audit.py`: append-only change events.

Endpoints fitting current `/warehouse/maps/{warehouse_map_id}` router style:

- `GET/PUT /warehouse/maps/{id}/coordinate-frame`
- `POST /warehouse/maps/{id}/coordinate-frame/validate`
- `POST /warehouse/maps/{id}/coordinate-frame/lock`
- `GET/POST /warehouse/maps/{id}/layout-versions`
- `GET /warehouse/maps/{id}/layout-versions/{version}`
- CRUD/batch endpoints for `/layout-versions/{version}/aisles`, `/racks`, `/shelves`, `/bins`, `/zones`, `/inspection-targets`
- `POST /warehouse/maps/{id}/layout-versions/{version}/validate`
- `POST /warehouse/maps/{id}/layout-versions/{version}/preview-mission`
- `POST /warehouse/maps/{id}/layout-versions/{version}/import?dry_run=true`
- `GET /warehouse/maps/{id}/layout-versions/{version}/export`
- `POST /warehouse/maps/{id}/structure/extract` should continue to fit existing architecture but return/progress toward a draft layout version, not only target rows.

Use PATCH where partial update semantics already exist. Prefer batch endpoints for shelves/bins to avoid hundreds of round trips. Every mutation accepts `If-Match`/revision and returns validation warnings. Keep existing scan-target endpoints during migration as compatibility projections over `InspectionTarget`.

## ROS / Robotics Improvements

- Rename canonical frames to `warehouse_map`, `odom`, `base_link`, `lidar_link`, `front_rgbd_link`, `front_rgbd_optical_frame`, `imu_link`, and gimbal/camera frames. Remap Gazebo-native names at bridge boundary.
- Add the localization component that publishes `warehouse_map -> odom`; SLAM owns this transform. Odometry owns `odom -> base_link`. Robot state/static publishers own body-to-sensor transforms.
- Ensure one broadcaster per edge. Extend readiness checks to detect duplicates, stale timestamps, cycles, and transform jumps.
- Generate static transforms from `WarehouseSensorRig` calibration/URDF, including camera optical rotations. Verify calibration hash before mapping.
- Require all PointCloud2/OccupancyGrid/mesh messages to carry explicit frames and timestamps. Transform at message time, not latest time, with bounded tolerance.
- Store occupancy origin quaternion and resolution; do not reduce occupancy origin to translation-only assumptions.
- Use `tf2_echo`, `view_frames`, `tf2_monitor`, RViz fixed-frame switching, and rosbag replay in diagnostics/runbooks.
- Feed current ESDF/occupancy into inspection preview and execution. Inflate obstacles by drone radius, localization uncertainty, downwash margin, and configured safety margin.
- Convert warehouse-map waypoints to control frame immediately before sending. Revalidate transform age/confidence for every segment or at a bounded cadence.
- Add a local-position MAVLink adapter if the autopilot executes indoor missions. Centralize ENU/NED and yaw conversion; do not route local warehouse poses through latitude/longitude `Coordinate` models.
- Validate Gazebo consistency with fixed landmark fixtures: world positions, ROS transformed positions, UI positions, and commanded vehicle positions must match within tolerance.

## Validation Rules

- All persisted layout coordinates reference a registered warehouse coordinate-frame version.
- Units are metres; angles are radians internally and explicitly suffixed degrees at API/UI edges.
- Warehouse boundary is valid, non-self-intersecting, and has floor/ceiling limits.
- Origin/coordinate version is locked before mission execution.
- Aisles remain inside boundary and retain usable width after margins.
- Rack belongs to an aisle and respects side/clearance rules.
- Shelf belongs to and is contained by a rack.
- Bin belongs to and is contained by a shelf.
- Objects do not overlap unless type-specific policy allows it.
- Inspection pose, approach corridor, hover volume, and exit segment have obstacle clearance.
- Complete waypoint path avoids occupied/unknown/no-fly volumes according to policy.
- Camera/gimbal line of sight reaches the target and pitch/yaw are within limits.
- Target height is within vehicle, sensor, ceiling, and downwash constraints.
- Dock/takeoff/landing approaches are reachable and clear.
- Map artifact, layout, frame, calibration, and mission versions match.
- Live transform is fresh, covariance is below threshold, and relocalization has not invalidated the plan.
- Coordinate values, dimensions, quaternions, normals, resolution, and scale are finite and normalized.
- Imports are dry-run validated; partial writes are transactional.

## Mission Planning Improvements

Generate semantic mission stages rather than four duplicate-pose action records:

1. localize at dock and verify frame/version;
2. takeoff to safe staging pose;
3. transit through aisle centerline/corridor;
4. approach rack face along validated normal;
5. settle at hover inspection pose;
6. aim gimbal/camera and verify line of sight;
7. capture barcode/QR/image/LiDAR result;
8. exit to aisle corridor before next target;
9. return to dock/home through validated route;
10. precision land or use emergency landing policy.

Aisle scans should produce layered serpentine paths within aisle volumes. Rack inspections should group by aisle/side and use safe face offsets. Shelf inspections should sequence vertically only when downwash and localization permit; otherwise return to corridor between levels. Bin-targeted missions should retain explicit target/approach/hover/exit poses.

Preview output must include transformed waypoint coordinates, yaw, gimbal pitch/yaw, path segments, clearance minimum, unknown-space exposure, estimated time/energy, stale-map warning, and return reserve. Execution must revalidate dynamic obstacles and current transforms. Abort behavior: stop/hover if safe, retreat to last aisle-safe pose, return to dock when localization/energy allows, otherwise land at validated emergency zone. Never continue with an unavailable or ambiguous map transform.

## Final Recommendations

First, restore the existing scan-target endpoints because the current naming regressions prevent reliable use or testing of the MVP. Second, stop treating `warehouse_map` and `odom` as interchangeable: add a persisted coordinate-frame version and a real `warehouse_map -> odom` localization contract. Third, make ESDF/occupancy path validation mandatory before any inspection mission can execute.

Only after those safety foundations should the team normalize aisle/rack/shelf/bin entities and expand the UI. The existing extraction engine, live Three.js viewer, local planner, and ROS point-cloud transforms are valuable building blocks; they should be retained, but connected through explicit frame/version/provenance contracts. This sequence limits migration risk while making coordinate setup repeatable, debuggable, and safe for real warehouse missions.
